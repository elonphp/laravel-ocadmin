# 行事曆管理

## 1. 設計目標

建立一套全系統共用的行事曆，定義每一天的工作日／休假類型。各公司可針對特定日期進行覆蓋，以因應不同的營運需求（例如 A 公司週六補班、B 公司不補班）。

## 2. 核心概念

```
系統行事曆（全域基準）
  ↓ 各公司可覆蓋特定日期
公司行事曆覆蓋
```

- **系統行事曆**：以年度為單位，定義每一天的日期類型（工作日、休息日、國定假日、補班日等）
- **公司覆蓋**：各公司僅記錄與系統行事曆不同的日期，查詢時優先使用公司覆蓋

## 3. 資料表設計

### 3.1 calendars（行事曆）

```
calendars
├── id
├── year            -- 年度（2025, 2026...）
├── name            -- 名稱（如「2026 台灣行事曆」）
├── is_default      -- 是否為該年度預設行事曆
├── is_active
└── timestamps
```

每年建立一份行事曆，作為全系統的基準。`is_default` 標記該年度生效的行事曆（同一年度僅一份為 default）。

### 3.2 calendar_days（行事曆日期）

```
calendar_days
├── id
├── calendar_id     -- FK → calendars
├── date            -- 日期（唯一，同一 calendar 內不重複）
├── day_type        -- 日期類型（見 §4）
├── description     -- 日期說明（來自政府資料，如「中秋節」、「補行上班日」）
├── note            -- 內部備註（人資自行註記用）
└── timestamps

UNIQUE(calendar_id, date)
```

**description vs note 區分**：
- `description`：來源為政府公開資料或系統產生的說明文字，描述該日期的官方定義（如「國慶日」、「農曆除夕」、「補行上班日」）
- `note`：人資人員手動填寫的內部備註（如「已確認」、「依公司公告調整」）

### 3.3 calendar_company_overrides（公司行事曆覆蓋）

```
calendar_company_overrides
├── id
├── company_id      -- FK → companies
├── date            -- 日期
├── day_type        -- 覆蓋後的日期類型
├── note            -- 備註／原因（如「工廠歲修」、「公司週年慶」）
└── timestamps

UNIQUE(company_id, date)
```

公司覆蓋表只記錄差異。未被覆蓋的日期，一律回退到系統行事曆。

## 4. 日期類型（day_type）

| 值 | 說明 | 通常對應 |
|----|------|---------|
| `workday` | 一般工作日 | 週一至週五 |
| `rest_day` | 休息日 | 週六 |
| `regular_day_off` | 例假日 | 週日 |
| `national_holiday` | 國定假日 | 中秋節、國慶日 |
| `makeup_workday` | 補班日 | 彈性放假後的週六補班 |
| `company_holiday` | 公司假日 | 公司週年慶（僅用於覆蓋表） |

以 Enum 實作（`App\Enums\Hrm\DayType`），方便程式引用與驗證。

### 4.1 一例一休（勞基法 §36）

2016 年修法後，每七日應有「一日例假」＋「一日休息日」，兩者在法律上為不同身分：

| 法律定義 | day_type | 能否出勤 | 加班費計算 |
|---------|----------|---------|-----------|
| **例假日** | `regular_day_off` | 原則不可，僅天災事變突發事件 | ×2 且須補假一天 |
| **休息日** | `rest_day` | 經勞工同意可出勤 | 前 2h ×1⅓, 3-8h ×1⅔, 9-12h ×2⅔（以 4h 為計算單位） |

其他日期類型的加班費率：

| day_type | 加班費率 | 說明 |
|----------|---------|------|
| `workday` | 前 2h ×1⅓, 之後 ×1⅔ | 一般工作日延長工時 |
| `national_holiday` | **×2** | 國定假日出勤，雙倍薪 |
| `makeup_workday` | 同 `workday` | 法律上視同工作日 |

> **重要**：`rest_day`（休息日）和 `regular_day_off`（例假日）的加班費率差異顯著，系統必須區分，薪資模組依此計算。

## 5. 查詢邏輯

判斷某員工某日的日期類型：

```
1. 查詢 calendar_company_overrides（員工所屬 company_id + date）
   → 有記錄 → 使用覆蓋的 day_type
   → 無記錄 ↓

2. 查詢 calendar_days（該年度 is_default 行事曆 + date）
   → 有記錄 → 使用系統行事曆的 day_type
   → 無記錄 ↓

3. 依星期幾判斷（週一～五 = workday，週六日 = rest_day）
```

第 3 步為最終 fallback，確保即使行事曆未建立完整也能正常運作。

## 6. 年度初始化

### 6.1 建立新年度行事曆

1. 建立 `calendars` 記錄（指定年度、名稱）
2. 自動產生 365/366 筆 `calendar_days`，預設依星期幾填入 `workday` 或 `rest_day`
3. 管理者手動調整國定假日與補班日

### 6.2 從上一年複製

提供「複製行事曆」功能，將上一年度的國定假日、補班日複製為新年度的草稿，再手動調整日期。

> 國定假日每年日期可能不同（如農曆節日），複製後仍需人工確認。

## 7. 與出勤系統的關係

行事曆是出勤判斷的基礎：

| 判斷 | 依據 |
|------|------|
| 今天是否為工作日 | 行事曆 day_type |
| 是否遲到／早退 | 工作日 + 班別時間 |
| 是否算加班 | 休假日出勤 or 工作日超時 |
| 請假扣除天數 | 只扣工作日 |

行事曆模組獨立於出勤模組，未來其他模組（如排班、薪資計算）也可引用。

## 8. 資料表關聯圖

```
calendars
└── calendar_days（該年度每一天的類型）

companies
└── calendar_company_overrides（公司專屬的日期覆蓋）

查詢某員工某日：
employee → company_id → overrides(date)
                          ↓ miss
                        calendar_days(date)
                          ↓ miss
                        weekday fallback
```

## 9. 政府公開行事曆資料來源

### 9.1 資料來源

台灣行政院人事行政總處（DGPA）每年公布「中華民國政府行政機關辦公日曆表」，可透過以下管道取得：

| 來源 | 格式 | 認證 | 說明 |
|------|------|------|------|
| **data.gov.tw** (dataset #14718) | CSV | 無 | 官方權威來源，每年一檔，4 欄位 |
| **data.taipei** | CSV | 無 | 台北市版本，5 欄位含 `holidayCategory`，較詳細 |
| **ruyut/TaiwanCalendar** (jsDelivr CDN) | JSON | 無 | 社群維護，格式乾淨，每年一檔 |

### 9.2 推薦方案：ruyut/TaiwanCalendar JSON

```
URL: https://cdn.jsdelivr.net/gh/ruyut/TaiwanCalendar/data/{year}.json
可用年度：2017 ~ 2026
```

JSON 結構（每年 365/366 筆，每天一筆）：

```json
[
  {"date": "20260101", "week": "四", "isHoliday": true,  "description": "開國紀念日"},
  {"date": "20260102", "week": "五", "isHoliday": false, "description": ""},
  {"date": "20260103", "week": "六", "isHoliday": true,  "description": ""},
  {"date": "20260208", "week": "六", "isHoliday": false, "description": "補行上班"},
  {"date": "20260217", "week": "二", "isHoliday": true,  "description": "春節"}
]
```

**欄位對應本系統 day_type 的轉換規則**：

| 優先序 | 條件 | 對應 day_type |
|--------|------|--------------|
| 1 | `isHoliday: false` 且 `description` 含「補行上班」或「調整上班」 | `makeup_workday` |
| 2 | `isHoliday: false` | `workday` |
| 3 | `isHoliday: true` 且 `description` 非空（如「中秋節」） | `national_holiday` |
| 4 | `isHoliday: true` 且 `description` 為空，**週日** | `regular_day_off`（例假日） |
| 5 | `isHoliday: true` 且 `description` 為空，**週六** | `rest_day`（休息日） |

> **補充**：`description` 在早期年度使用「調整上班」，2021 年後改為「補行上班」，程式需同時比對。
>
> 政府資料的 `isHoliday` 欄位僅表示「當日是否放假」，並未標註該日屬於勞基法的「休息日」或「例假日」。但兩者的加班費率不同（見 §4.1），系統在匯入時依星期幾自動對應：週六 → `rest_day`（休息日）、週日 → `regular_day_off`（例假日）。若企業的例假日／休息日安排與週六日不同，匯入後可手動調整。

### 9.3 匯入流程設計

```
管理介面：行事曆表單 → [匯入政府行事曆] 按鈕

流程：
1. 依行事曆的 year 欄位，從 CDN 下載該年度 JSON
2. 解析每筆資料，轉換為 day_type + description
3. 批次 upsert 至 calendar_days（以 calendar_id + date 為 unique key）
4. 回報匯入結果（新增 N 筆、更新 M 筆）
```

後端路由：`POST hrm/calendar/{calendar}/import-government`

### 9.4 為什麼不另建政府行事曆資料表

政府行事曆資料本質上就是系統行事曆的初始資料來源。另建資料表會產生：
- 兩份資料的同步問題
- 查詢時需額外 JOIN 或 merge 邏輯
- 增加維護複雜度

正確做法是將政府資料直接匯入 `calendar_days`，透過 `description` 欄位保留原始說明。人資可在匯入後手動微調（例如修改 day_type 或補充 note），不影響已匯入的 description。

### 9.5 備用方案：data.taipei CSV

若 CDN JSON 不可用，可改用台北市政府開放資料 CSV：

```
URL: https://data.taipei/api/frontstage/tpeod/dataset/resource.download?rid=0dcbcfcf-f7a1-4664-a810-82c01cb524e0
格式：CSV（單一檔案涵蓋多年度）
欄位：Date, name, isHoliday, holidayCategory, description
```

`holidayCategory` 提供更細的分類（放假之紀念日及節日、補假、調整放假日、補行上班日等），可做為更精確的 day_type 轉換依據。
