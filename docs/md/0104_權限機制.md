## 一、概述

### 1.1 技術選型

採用 [Spatie Laravel Permission](https://spatie.be/docs/laravel-permission) 作為權限管理基礎。

| 項目 | 說明 |
|------|------|
| 套件 | spatie/laravel-permission ^6.24 |
| Guard | 單一 Guard (`web`) |
| 角色命名 | Portal 前綴：`{portal}.{role_name}`（`super_admin` 例外） |
| 權限命名 | 四段式：`{portal}.{module}.{resource}.{action}` |
| 萬用權限 | Spatie Wildcard Permission（`enable_wildcard_permission => true`） |
| 多語支援 | 角色與權限的 `display_name`、`note` 透過翻譯表實現 |
| 資料表前綴 | `acl_`（透過 `config/permission.php` 設定） |

### 1.2 權限類型

| 類型 | 說明 | 實現方式 |
|------|------|----------|
| **功能權限** | 能否執行某操作 | Spatie Permission |
| **資料權限** | 能看到哪些資料 | 組織層級 + Policy |

### 1.3 現有基礎設施

- [x] `spatie/laravel-permission` 套件安裝
- [x] `config/permission.php` 設定（`acl_` 前綴、wildcard 啟用）
- [x] ACL 資料庫遷移（含翻譯表）
- [x] 自訂 Model：`App\Models\Acl\Role`、`App\Models\Acl\Permission`（含多語翻譯）
- [x] `User` Model 已掛載 `HasRoles` trait
- [x] 角色 Seeder (`AclRoleSeeder`)
- [ ] 權限 Seeder
- [ ] 路由級權限檢查

## 二、角色設計

### 2.1 命名規範

**格式：`{portal}.{role_name}`**

```
hrm.hr_manager
 │      │
 │      └── 角色名稱（snake_case）
 └── Portal 前綴
```

例外：`super_admin` 為全域角色，不帶 Portal 前綴。

> Portal 的定義與架構決策請見 [0105_Portal概述.md](0105_Portal概述.md)。

### 2.2 角色清單

#### 全域角色

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `super_admin` | 超級管理員 | 最高權限，僅用於 Ocadmin |

#### HRM Portal 角色（hrm.*）

**HR 管理角色：**

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `hrm.hr_manager` | HR 主管 | 管理所有 HR 功能，含薪資、報表 |
| `hrm.hr_operator` | HR 管理員 | 日常 HR 操作：員工資料、出勤、請假 |
| `hrm.payroll_manager` | 薪資主管 | 薪資計算、薪資單管理 |

**員工自助服務角色：**

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `hrm.dept_manager` | 部門主管 | 審核下屬請假/出勤，檢視部門資料 |
| `hrm.employee` | 一般員工 | 查看個人資料、打卡、申請請假 |

#### Ocadmin Portal 角色（admin.*）

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `admin.store_manager` | 門市店長 | 擁有大部分 admin.* 權限 |
| `admin.operator` | 後台操作員 | 日常後台操作 |
| `admin.viewer` | 唯讀檢視者 | 僅擁有 `*.access` 權限 |

### 2.3 跨角色指派

同一帳號可擁有多個角色：

```php
// HR 主管：管理功能 + 自己也是員工
$user->roles: ['hrm.hr_manager', 'hrm.employee']

// 部門主管：ESS + 部門管理
$user->roles: ['hrm.dept_manager', 'hrm.employee']

// 超級管理員：僅限 Ocadmin
$user->roles: ['super_admin']
```

## 三、權限設計

### 3.1 命名規範

**格式：`{portal}.{module}.{resource}.{action}`**

- **原則上 4 段**，使用點分隔，全小寫，snake_case
- resource 一律使用**單數**（如 `product`、`order`、`payment`，不用 `products`）
- module 多層用底線連接

| 段落 | 說明 | 範例 |
|------|------|------|
| `portal` | 系統入口 | `hrm`, `admin` |
| `module` | 功能模組 | `ess`（自助）, `mss`（管理）, `team`（部門主管）, `catalog`, `order` |
| `resource` | 資源名稱 | `profile`, `employee`, `product`, `order`, `role` |
| `action` | 操作動作 | `access`, `modify`, `delete`, `approve`, `export` |

**非四段的例外情況：**

功能權限（CRUD 操作）固定 4 段，但用於**選單群組可見性控制**時，可使用較短的格式：

| 段數 | 格式 | 用途 | 範例 |
|------|------|------|------|
| 4 段 | `{portal}.{module}.{resource}.{action}` | 功能權限（原則） | `admin.master.supplier.access` |
| 2 段 | `{portal}.{module}` | 模組入口可見性 | `admin.master` |
| 3 段 | `{portal}.{module}.{resource}` | 資源入口可見性 | `admin.master.supplier` |

例如「主檔管理」選單群組設定 `permission: admin.master.access`，沒有此權限的角色整個主檔管理選單不可見，即使擁有其下子項的功能權限（見 §12.4 父層可見性規則）。

### 3.2 萬用權限（Wildcard）

啟用 Spatie 原生 wildcard permission（`config/permission.php` 中 `enable_wildcard_permission => true`）。

使用 `*` 作為萬用字元：

```
# 單一資源的所有動作
hrm.mss.employee.*        → 符合 hrm.mss.employee.access、.modify、.delete、.export

# 整個模組的所有權限
hrm.mss.*.*               → 符合 hrm.mss.employee.*、hrm.mss.leave.*、hrm.mss.holiday.*

# 整個 Portal 的所有權限
hrm.*.*.*                  → 符合所有 hrm 權限

# 明確指定單一動作
hrm.ess.leave.modify       → 僅符合「請假申請」
```

> **注意**：不要建立 `*.all` 權限，使用 Spatie 原生 wildcard `*` 即可達成相同效果，無需額外的 Gate 邏輯。

### 3.4 常用 Action

| Action | 說明 | 涵蓋範圍 |
|--------|------|----------|
| `access` | 檢視 | 列表 + 單筆檢視（原 list + read） |
| `modify` | 修改 | 新增 + 修改（原 create + update） |
| `delete` | 刪除 | |
| `export` | 匯出 | |
| `approve` | 審核 | 適用於請假、出勤、退款等需審核流程 |

> **簡化原則**：能看就能列表（access），能改就能建（modify），刪除與審核獨立控管。

| 資料表 | 說明 |

| `acl_model_has_permissions` | 使用者-權限關聯（直接指派） |

### 4.3 擴展欄位

`acl_permissions` 額外欄位：

| 欄位 | 類型 | 說明 |
|------|------|------|

| 欄位 | 類型 | 說明 |
|------|------|------|


### 5.2 自訂 Model

角色和權限 Model 繼承自 Spatie 基礎類別，加上 `HasTranslation` trait 實現多語支援：

```php
// app/Models/Acl/Role.php
class Role extends SpatieRole
{
    use HasTranslation;

    protected $fillable = ['name', 'guard_name', 'sort_order', 'is_active'];
    protected array $translatedAttributes = ['display_name', 'note'];
    protected string $translationModel = RoleTranslation::class;
}

// app/Models/Acl/Permission.php
class Permission extends SpatiePermission
{
    use HasTranslation;

    protected $fillable = [
        'parent_id', 'name', 'guard_name',
        'type', 'icon', 'sort_order', 'is_active',
    ];

    protected array $translatedAttributes = ['display_name', 'note'];
    protected string $translationModel = PermissionTranslation::class;

    // 樹狀結構與選單相關方法見 §8.5
}
```

### 5.3 User Model

```php
// app/Models/User.php
use Spatie\Permission\Traits\HasRoles;

class User extends Authenticatable
{
    use HasRoles;

    /**
     * 檢查使用者是否擁有指定 Portal 的角色
     */
    public function hasPortalRole(string $portal): bool
    {
        if ($this->hasRole('super_admin')) {
            return true;
        }

        return $this->roles->contains(
            fn ($role) => str_starts_with($role->name, $portal . '.')
        );
    }
}
```

## 六、權限檢查

### 6.1 Controller 中

```php
public function index()
{
    // 方式一：手動檢查
    if (!auth()->user()->can('hrm.mss.employee.access')) {
        abort(403);
    }

    // 方式二：authorize（拋出 AuthorizationException）
    $this->authorize('hrm.mss.employee.access');
}
```

### 6.2 Middleware

```php
// 路由中使用
Route::middleware(['permission:hrm.mss.employee.access'])->group(function () {
    Route::get('/employees', [EmployeeController::class, 'index']);
});

// 或
Route::get('/employees', [EmployeeController::class, 'index'])
    ->middleware('can:hrm.mss.employee.access');
```

### 6.3 Inertia + React 前端

後端透過 Inertia 傳遞權限資訊給前端：

```php
// app/Http/Middleware/HandleInertiaRequests.php

public function share(Request $request): array
{
    return [
        ...parent::share($request),
        'auth' => [
            'user' => $request->user(),
            'permissions' => $request->user()?->getAllPermissions()
                ->pluck('name')
                ->toArray() ?? [],
            'roles' => $request->user()?->getRoleNames()->toArray() ?? [],
        ],
    ];
}
```

React 前端使用：

```tsx
// 取得權限資訊
const { permissions } = usePage().props.auth;

// 權限檢查 helper
function can(permission: string): boolean {
    return permissions.includes(permission);
}

// 在 JSX 中使用
{can('hrm.mss.employee.modify') && (
    <Button>新增員工</Button>
)}
```

### 6.4 欄位級權限

```php
// Controller
public function show(Employee $employee)
{
    $data = $employee->toArray();

    // 隱藏無權限的敏感欄位
    if (!auth()->user()->can('hrm.mss.payroll.access:salary')) {
        unset($data['salary']);
    }
    if (!auth()->user()->can('hrm.mss.payroll.access:bonus')) {
        unset($data['bonus']);
    }

    return Inertia::render('Employee/Detail', ['employee' => $data]);
}
```

### 7.1 Policy

```php
// app/Policies/EmployeePolicy.php

class EmployeePolicy
{
    public function viewAny(User $user): bool
    {
        return $user->can('hrm.mss.employee.access')
            || $user->can('hrm.team.employee.access');
    }

    public function view(User $user, Employee $employee): bool
    {
        // HR 有權限可查看所有
        if ($user->can('hrm.mss.employee.access')) {
            return true;
        }

        // 查看自己
        if ($employee->user_id === $user->id) {
            return true;
        }

        // 部門主管查看下屬
        if ($user->can('hrm.team.employee.access')) {
            return $employee->department_id === $user->employee?->department_id;
        }

        return false;
    }
}
```

## 八、選單系統（DB 驅動方案，目前未採用）

> **注意：** 本節描述的是將選單寫入 `acl_permissions` 表的 DB 驅動方案。經評估後決定採用**明碼拆檔 + 權限過濾**（見 §12），選單由各模組的 Menu Class 定義，不從 DB 撈取。本節保留作為備案參考，若未來需要「後台 UI 管理選單」可回頭採用。

### 8.1 概念

選單與權限共用同一張 `acl_permissions` 資料表，透過 `type` 欄位區分用途：

| type | 用途 | 層級 | 範例 |
|------|------|------|------|
| `menu` | 選單節點 | L1 ~ L3 | `hrm`, `hrm.mss`, `hrm.mss.employee` |
| `action` | 功能權限 | L4 | `hrm.mss.employee.access`, `hrm.mss.employee.modify` |

**關鍵概念：**

1. `acl_permissions.type=menu` 的記錄列舉了全系統所有選單項目（sidebar 的完整結構）
2. 使用者能否看到某個選單，依 `acl_role_has_permissions` 或 `acl_model_has_permissions` 判斷 — 有該筆 menu permission 才顯示

menu 和 action 都是 `acl_permissions` 表裡的記錄，授權機制完全相同。`type` 欄位**僅供分類參考**，Spatie 的 `hasPermissionTo()` / `can()` 只比對 `name`，不看 `type`。

### 8.2 樹狀結構

權限表同時存在 1～4 段的記錄，透過 `parent_id` 構成選單樹：

```
hrm                              (L1, type=menu, parent_id=null)
├── hrm.mss                      (L2, type=menu, parent_id → hrm)
│   ├── hrm.mss.employee         (L3, type=menu, parent_id → hrm.mss)
│   │   ├── hrm.mss.employee.access   (L4, type=action)
│   │   ├── hrm.mss.employee.modify   (L4, type=action)
│   │   └── hrm.mss.employee.delete   (L4, type=action)
│   └── hrm.mss.leave            (L3, type=menu, parent_id → hrm.mss)
│       ├── hrm.mss.leave.access       (L4, type=action)
│       └── hrm.mss.leave.approve      (L4, type=action)
└── hrm.ess                      (L2, type=menu, parent_id → hrm)
    └── hrm.ess.profile          (L3, type=menu, parent_id → hrm.ess)
        ├── hrm.ess.profile.access     (L4, type=action)
        └── hrm.ess.profile.modify     (L4, type=action)
```

> menu 節點（L1～L3）與 action 節點（L4）的 `name` 天然不同（段數不同），不會衝突。

### 8.3 選單產生流程

```
acl_permissions（type=menu）    使用者的權限
        │                           │
        ▼                           ▼
   完整選單樹          ∩        使用者權限集合
        │
        ▼
   使用者可見選單
```

1. 從 `acl_permissions` 撈 `type=menu` → **完整選單樹**
2. 與使用者擁有的權限取交集 → **使用者可見選單**
3. 使用者未擁有的 menu 權限 → 該選單項目隱藏

### 8.4 自動指派父層權限

指派 L4 action 權限時，必須同時指派其 L1～L3 menu 權限，否則選單路徑不完整、使用者看不到入口。

```php
// Role Model
public function assignPermissionWithAncestors(Permission $permission)
{
    // 指派權限本身
    if (!$this->hasPermissionTo($permission)) {
        $this->givePermissionTo($permission);
    }

    // 自動指派所有父層權限（確保選單路徑完整）
    foreach ($permission->ancestors() as $ancestor) {
        if (!$this->hasPermissionTo($ancestor)) {
            $this->givePermissionTo($ancestor);
        }
    }
}
```

範例：指派 `hrm.mss.employee.access` 時，自動一併指派：
- `hrm.mss.employee`（L3, menu）
- `hrm.mss`（L2, menu）
- `hrm`（L1, menu）

### 8.5 Permission Model 擴展

```php
// app/Models/Acl/Permission.php

class Permission extends SpatiePermission
{
    use HasTranslation;

    // 父層權限
    public function parent(): BelongsTo
    {
        return $this->belongsTo(Permission::class, 'parent_id');
    }

    // 子層權限
    public function children(): HasMany
    {
        return $this->hasMany(Permission::class, 'parent_id')
            ->orderBy('sort_order');
    }

    // 取得所有祖先（遞迴向上）
    public function ancestors(): Collection
    {
        $ancestors = collect();
        $parent = $this->parent;
        while ($parent) {
            $ancestors->push($parent);
            $parent = $parent->parent;
        }
        return $ancestors;
    }

    // 是否為選單項目
    public function isMenu(): bool
    {
        return $this->type === 'menu';
    }

    // 是否為功能權限
    public function isAction(): bool
    {
        return $this->type === 'action';
    }
}
```

## 十一、使用者權限快取

### 11.1 問題

Spatie 內建快取僅快取**全域權限定義表**（所有 permission 與 role 的對應關係），不快取個別使用者的權限。每個 request 仍需：

1. 載入使用者的 roles（1 次 DB query）
2. 載入使用者的 direct permissions（1 次 DB query）
3. In-memory 比對

系統規模擴大（使用者多、權限多）時，每個 request 固定 2 次 DB query 可優化。

### 11.2 方案：角色組合快取 + 全域版本號

**核心概念：** 將使用者的 role ids 排序後組成 cache key，快取該角色組合的所有權限名稱。相同角色組合的使用者共用同一份快取。

**Cache 結構：**

```
┌─────────────────────────┐     ┌──────────────────────────────────┐
│ key: role_perm_ver      │     │ key: role:1-3-11:v42             │
│ val: 42                 │     │ val: ['admin.catalog.product.ac… │
│ TTL: forever            │     │       'admin.master.supplier.ac… │
│                         │     │       ...]                       │
│ （全域版本號，所有角色   │     │ TTL: 7 days                      │
│   組合共用一個）         │     │                                  │
└─────────────────────────┘     │ （角色 1+3+11 的所有權限）        │
                                └──────────────────────────────────┘
```

### 11.3 查詢流程

```
使用者發 request
    │
    ├─ 1. 取 user 的 role_ids → [1, 3, 11]（從 DB 或 session）
    │
    ├─ 2. 取全域版本號 → Cache::get('role_perm_ver') → 42
    │
    ├─ 3. 組 cache key → "role:1-3-11:v42"
    │
    ├─ 4. Cache::get("role:1-3-11:v42")
    │     ├─ hit  → 拿到權限陣列（0 DB query）
    │     └─ miss → 查 DB 建立權限陣列，寫入 cache（TTL 7 天）
    │
    └─ 5. $user->can() 在 in-memory 比對
```

**每 request 成本：** 2 次 cache get（版本號 + 權限），cache hit 時 0 次 DB query。

### 11.4 快取失效：全域版本號

角色的權限被修改時，**只遞增版本號，不主動重建快取**：

```php
// 角色權限變更時（如 RoleController 儲存權限後）
Cache::increment('role_perm_ver');
```

- 版本號從 42 → 43
- 所有 request 組出的 key 自動從 `v42` 變 `v43`
- 全部 cache miss，各自在下一次 request 時從 DB 重建
- 舊版 `v42` 的 key 無人再讀，靠 TTL（7 天）自然回收

**不在變更當下重建的原因：** 管理員設定權限時可能連續修改多次，若每次都重建會造成不必要的 IO。Lazy rebuild（等 request 進來時才建）更合理。

### 11.5 實作要點

**User Model 覆寫 `can()`：**

```php
// app/Models/User.php

public function can($ability, $arguments = [])
{
    // super_admin 跳過快取
    if ($this->hasRole('super_admin')) {
        return true;
    }

    $permissions = $this->getCachedPermissions();

    return in_array($ability, $permissions);
}

protected function getCachedPermissions(): array
{
    $roleIds = $this->roles->pluck('id')->sort()->implode('-');
    $ver = Cache::get('role_perm_ver', 1);
    $key = "role:{$roleIds}:v{$ver}";

    return Cache::remember($key, now()->addDays(7), function () {
        return $this->getPermissionsViaRoles()
            ->pluck('name')
            ->toArray();
    });
}
```

**角色權限變更時遞增版本號：**

```php
// 角色權限 sync 後
$role->syncPermissions($permissions);
Cache::increment('role_perm_ver');
```

**版本號初始化（AppServiceProvider boot）：**

```php
// Cache::add 僅在 key 不存在時寫入，每次 boot 呼叫無副作用
Cache::add('role_perm_ver', 1);
```

### 11.6 TTL 設計

| Cache entry | TTL | 原因 |
|-------------|-----|------|
| `role_perm_ver`（版本號） | forever | 關鍵資料，不可意外過期 |
| `role:{ids}:v{ver}`（權限集合） | 7 天 | 當前版本實質 forever（持續命中）；舊版靠 TTL 自動回收 |

### 11.7 注意事項

- **直接指派權限（`model_has_permissions`）：** 此快取僅涵蓋透過角色取得的權限。若有使用直接指派，需在 cache key 中加入 user id 或另行處理
- **角色變更 vs 權限變更：** 使用者的角色被更換時，其 role_ids 組合改變，自然組出不同 cache key，不需要額外處理版本號
- **Cache driver：** 此方案不依賴 Cache Tags，任何 driver（file、database、Redis、Memcached）皆可使用
- **停用帳號：** 緊急封鎖惡意使用者應透過停用帳號（`is_active = false`，auth middleware 攔截），不應依賴權限快取失效

### 11.8 與 Spatie 內建快取的比較

| | Spatie 內建 | 角色組合快取 |
|---|---|---|
| **快取內容** | 全域權限定義表 | 角色組合 → 權限名稱集合 |
| **每 request DB query** | 2 次（user→roles, user→permissions） | 0 次（cache hit 時） |
| **快取命中率** | 100%（全域一份） | 高（同角色組合共用，實務上組合有限） |
| **失效機制** | 權限/角色定義異動時自動清 | 全域版本號遞增 |
| **Cache driver 限制** | 無 | 無 |

## 十二、選單與權限整合

### 12.1 需求

使用者只應看到自己有權限的選單項目。點進無權限的頁面會被擋 403，但選單上不應出現這些入口。

### 12.2 方案比較

#### 方案 A：明碼拆檔 + 權限過濾（推薦）

選單以 PHP 檔案定義，每個模組一個 Menu class，MenuComposer 收集後依使用者權限過濾。

```
app/Portals/Ocadmin/
├── Core/ViewComposers/MenuComposer.php    ← 收集 + 權限過濾
├── Modules/
│   ├── Catalog/CatalogMenu.php            ← 商品型錄
│   ├── Inventory/InventoryMenu.php        ← 庫存管理
│   ├── Master/MasterMenu.php              ← 主檔管理
│   ├── Finance/FinanceMenu.php            ← 財務會計
│   └── Hrm/HrmMenu.php                   ← 人事管理
```

#### 方案 B：選單寫入 permissions 表（§8 樹狀結構）

選單節點以 `type=menu` 存入 `acl_permissions` 表，與 action 權限共用同一張表和 `parent_id` 樹狀結構。MenuComposer 改從 DB 撈取 `type=menu`，與使用者權限取交集產生可見選單。

#### 比較

| | A. 明碼拆檔 | B. 寫入 DB |
|---|---|---|
| **選單定義位置** | PHP 檔案（版控追蹤） | `acl_permissions` 表 |
| **修改方式** | 改 code + deploy | 改 DB（後台 UI 或 seeder） |
| **可讀性 / debug** | 打開檔案就看到 | 要查 DB，parent_id 樹狀不直覺 |
| **DB query** | 0（純 PHP array） | 1 次（可快取） |
| **Seeder 維護** | 不需要 | 需維護 parent_id 對應關係 |
| **後台 UI 管理選單** | 不支援 | 支援 |
| **新增模組** | 寫 Menu class + 註冊 | 寫 seeder + 跑 seed |
| **適合場景** | 選單結構穩定、開發團隊維護 | 選單需由非工程人員管理 |

#### 結論

**採用方案 A。** 內部後台的選單結構穩定，開發階段改 code 比維護 seeder 更直覺。方案 B 的核心優勢「後台 UI 管理選單」在目前階段不需要。未來如有需求，方案 A 的 Menu array 可直接搬入 seeder，升級路徑自然。

### 12.4 MenuComposer 收集與過濾

```php
// app/Portals/Ocadmin/Core/ViewComposers/MenuComposer.php

use App\Portals\Ocadmin\Modules\Catalog\CatalogMenu;
use App\Portals\Ocadmin\Modules\Inventory\InventoryMenu;
use App\Portals\Ocadmin\Modules\Master\MasterMenu;

class MenuComposer
{
    /**
     * 選單註冊表 — 順序即為 sidebar 顯示順序
     */
    protected array $menuClasses = [
        // DashboardMenu::class,
        CatalogMenu::class,
        InventoryMenu::class,
        MasterMenu::class,
        // MemberMenu::class,
        // FinanceMenu::class,
        // SystemMenu::class,
    ];

    public function compose(View $view): void
    {
        $view->with('menus', $this->buildMenus());
    }

    protected function buildMenus(): array
    {
        $user = auth()->user();

        return collect($this->menuClasses)
            ->map(fn ($class) => $class::items())
            ->map(fn ($menu) => $this->filterByPermission($menu, $user))
            ->filter()         // 移除 null（整個群組無權限）
            ->values()
            ->toArray();
    }

    /**
     * 遞迴過濾：移除無權限的項目，子項全空的父層也移除
     */
    protected function filterByPermission(array $item, $user): ?array
    {
        // 檢查項目本身的權限
        if (!empty($item['permission']) && !$user->can($item['permission'])) {
            return null;
        }

        // 遞迴過濾子項目
        if (!empty($item['children'])) {
            $item['children'] = collect($item['children'])
                ->map(fn ($child) => $this->filterByPermission($child, $user))
                ->filter()
                ->values()
                ->toArray();

            // 分組節點（href 為空）且子項全被過濾 → 隱藏父層
            if (empty($item['children']) && empty($item['href'])) {
                return null;
            }
        }

        return $item;
    }
}
```

**過濾邏輯：**

1. 項目有 `permission` 欄位 → 使用者無此權限則移除
2. 項目無 `permission` 欄位 → 不檢查，直接保留
3. 子項目全部被移除且父層 `href` 為空 → 父層也移除（避免空的分組節點）
4. `super_admin` 的 `can()` 永遠回傳 `true`（§11.5 User Model 已處理），選單全部可見

**父層可見性規則：** 父層選單（如「系統管理」、「財務會計」）通常不設 `permission`，其可見性由子項向上推導決定：

- 至少一個子項有權限 → 父層**顯示**
- 所有子項都無權限 → 子項全被過濾 → 父層自動**隱藏**

因此不需要為父層選單額外定義選單專用權限（如 `admin.system.menu`），子項的 action 權限（如 `admin.system_access.user.access`）已足夠控制整個選單群組的可見性。

### 12.5 新增模組 Checklist

新增一個功能模組的選單項目：

- [ ] 在模組目錄下建立 `{Module}Menu.php`，實作 `items()` 方法
- [ ] 每個子項目填入對應的 `permission`（通常是 `access` 層級）
- [ ] 在 `MenuComposer::$menuClasses` 中註冊，插入期望的排序位置
- [ ] 確認對應的權限已在 `AclPermissionSeeder` 中定義

## 十三、總結

### 13.1 快速參考

| 項目 | 格式 | 範例 |
|------|------|------|
| 全域角色 | `{role}` | `super_admin` |
| Portal 角色 | `{portal}.{role}` | `hrm.hr_manager`, `admin.operator` |
| 權限 | `{portal}.{module}.{resource}.{action}` | `hrm.mss.employee.access` |
| 萬用權限 | `{portal}.{module}.{resource}.*` | `hrm.mss.employee.*` |
| 欄位權限 | `{portal}.{module}.{resource}.{action}:{column}` | `hrm.mss.payroll.access:salary` |

### 13.2 檢查方式

```php
// 檢查角色
$user->hasRole('super_admin');
$user->hasRole('hrm.hr_manager');

// 檢查權限（支援 wildcard 比對）
$user->can('hrm.mss.employee.access');

// 檢查 Portal 存取（見 0105_Portal概述.md）
$user->hasPortalRole('hrm');
$user->hasPortalRole('admin');
```

## 十四、單控制器多選單入口

### 14.1 情境

同一筆資料（如供應商）需要從不同選單入口操作：

- **主檔管理 > 供應商**：`/admin/master/suppliers`
- **庫存管理 > 供應商**：`/admin/inventory/suppliers`

兩個入口使用**同一個 Controller**（`SupplierController`），但使用者從哪個入口進來，返回按鈕、列表連結、分頁連結就要指回對應的路徑。

### 14.2 路由：相同 Controller 掛在不同前綴

```php
// 主檔管理 > 供應商
Route::prefix('master')->name('master.')->group(function () {
    Route::prefix('suppliers')->name('supplier.')->group(function () {
        Route::get('/', [SupplierController::class, 'index'])->name('index');
        Route::get('/list', [SupplierController::class, 'list'])->name('list');
        Route::get('/create', [SupplierController::class, 'create'])->name('create');
        // ...
    });
});

// 庫存管理 > 供應商（共用同一個 Controller）
Route::prefix('inventory')->name('inventory.')->group(function () {
    Route::prefix('suppliers')->name('supplier.')->group(function () {
        Route::get('/', [SupplierController::class, 'index'])->name('index');
        Route::get('/list', [SupplierController::class, 'list'])->name('list');
        Route::get('/create', [SupplierController::class, 'create'])->name('create');
        // ...
    });
});
```

產生兩組 route name：

| 入口 | route name 前綴 |
|------|----------------|
| 主檔管理 | `lang.ocadmin.master.supplier` |
| 庫存管理 | `lang.ocadmin.inventory.supplier` |

### 14.3 Controller：用 `routePrefix()` 動態推算

Controller 不寫死路由名稱，而是從當前 request 的 route name 反推前綴：

```php
/**
 * 從當前 route name 反推前綴
 * e.g. 'lang.ocadmin.master.supplier.index' → 'lang.ocadmin.master.supplier'
 *      'lang.ocadmin.inventory.supplier.edit' → 'lang.ocadmin.inventory.supplier'
 */
protected function routePrefix(): string
{
    return preg_replace('/\.[^.]+$/', '', request()->route()->getName());
}
```

所有 URL 都透過前綴動態產生：

```php
public function index(Request $request): View
{
    $prefix = $this->routePrefix();

    $data['list_url']  = route("{$prefix}.list");
    $data['index_url'] = route("{$prefix}.index");
    $data['add_url']   = route("{$prefix}.create");
    // ...
}

public function create(): View
{
    $prefix = $this->routePrefix();

    $data['save_url'] = route("{$prefix}.store");
    $data['back_url'] = route("{$prefix}.index");  // 返回按鈕回到來源列表
    // ...
}
```

### 14.4 View：不需任何修改

View 只使用 Controller 傳入的 `$back_url`、`$list_url` 等變數，不關心具體路徑。

### 14.5 效果

| 使用者操作 | 產生的 URL |
|-----------|-----------|
| 從主檔管理進入，點新增 | `/admin/master/suppliers/create` |
| 按返回 | `/admin/master/suppliers` |
| 從庫存管理進入，點新增 | `/admin/inventory/suppliers/create` |
| 按返回 | `/admin/inventory/suppliers` |

### 14.6 注意事項

- 兩組路由的 action name（`index`、`create`、`store`、`edit`、`update`、`destroy`）必須一致
- Route Model Binding 的參數名也要一致（如 `{organization}`）
- 權限控制在 Controller constructor 中處理，兩個入口可共用或分別設定權限
