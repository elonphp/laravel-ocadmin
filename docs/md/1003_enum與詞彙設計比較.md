# Enum 與詞彙表（Lookup Table）設計比較

## 問題

狀態類欄位（如訂單狀態）該用 `INT` 外鍵關聯詞彙表，還是用 `VARCHAR` 搭配 Laravel Enum？

## 儲存空間

以 `orders.status` 為例：

| 方式 | 型態 | 單筆大小 |
|------|------|----------|
| INT FK（如 OpenCart） | `INT` | 4 bytes |
| VARCHAR + Enum | `VARCHAR(20)` | ~20 bytes |

差異：每筆約 16 bytes。**一萬筆訂單僅多 ~0.15 MB**，幾乎可忽略。

## 兩種方式的比較

### INT 外鍵 + 詞彙表（Lookup Table）

**適用場景：**
- 狀態需要多語系顯示
- 狀態需要讓管理員在後台自訂、動態增減
- 不想因為新增狀態而改程式碼

**典型欄位：** `order_status`、`product_category`、`shipping_method`

**優點：**
- 支援多語系（詞彙表可存多語言名稱）
- 狀態名稱可動態修改

**缺點：**
- 查詢需要 JOIN
- 程式碼中用數字 ID，可讀性差

### VARCHAR + Laravel Enum

**適用場景：**
- 狀態由開發者定義，業務邏輯固定
- 狀態數量少且穩定
- 不需要後台動態管理

**典型欄位：** `payment_status`（pending/paid/refunded）、`source_type`（PUR/MFG）

**優點：**
- 不需要 JOIN，查詢簡單
- 程式碼可讀性高，有型別安全
- 資料庫值直接看得懂，除錯方便

**缺點：**
- 新增狀態需要改程式碼並部署
- 多語系需透過語言檔處理

## 真正的效能瓶頸

現代硬碟便宜，欄位型態造成的空間差異不是瓶頸。真正影響效能的是：

1. **缺少索引** — 影響遠大於多幾個 VARCHAR 欄位
2. **N+1 查詢** — 比空間更值得關注
3. **沒有 Cache** — 影響遠大於欄位型態選擇

## 結論：混合策略

根據欄位性質決定，不必全部統一：

| 判斷條件 | 選擇 |
|----------|------|
| 需要後台動態管理或多語系 | INT FK + 詞彙表 |
| 開發者定義、邏輯固定、數量穩定 | Laravel Enum |

多幾個 Enum 欄位加起來可能多幾 KB～幾 MB，換來的是程式碼清晰、型別安全、不需要多餘的 JOIN。這筆交換在大多數專案裡非常划算。
