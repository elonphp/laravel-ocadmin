## 一、概述

後台側欄選單需依使用者權限動態顯示：使用者只能看到自己有權限的選單項目。本文件記錄兩種選單方案的完整設計，不論最終採用哪一種，兩者皆保留供隨時參考。

- **方案 A — Code Driven**：選單以 PHP Menu Class 定義，MenuComposer 收集後依權限過濾。
- **方案 B — DB Driven**：選單存入獨立的 `sys_menus` / `sys_menu_translations` 表，與權限表分離，透過 `permission_name` 欄位關聯權限做可見性判斷。

兩種方案的優劣比較與結論見 [§4 方案評估比較](#四方案評估比較)。

---

## 二、方案 A — Code Driven（MenuComposer）

選單以 PHP 檔案定義，每個模組一個 Menu class，MenuComposer 收集後依使用者權限過濾。

```
app/Portals/Ocadmin/
├── Core/ViewComposers/MenuComposer.php    ← 收集 + 權限過濾
├── Modules/
│   ├── Catalog/CatalogMenu.php            ← 商品型錄
│   ├── Inventory/InventoryMenu.php        ← 庫存管理
│   ├── Master/MasterMenu.php              ← 主檔管理
│   ├── Finance/FinanceMenu.php            ← 財務會計
│   └── Hrm/HrmMenu.php                   ← 人事管理
```

### 2.1 Menu Class 規範

每個模組在自己的目錄下定義 Menu class，實作 `items()` 靜態方法，回傳單一選單群組：

```php
// app/Portals/Ocadmin/Modules/Finance/FinanceMenu.php

namespace App\Portals\Ocadmin\Modules\Finance;

class FinanceMenu
{
    public static function items(): array
    {
        return [
            'id'         => 'menu-finance',
            'icon'       => 'fa-solid fa-coins',
            'name'       => '財務會計',
            'permission' => 'admin.finance',       // 父層權限（可省略，有子項即顯示）
            'href'       => '',
            'children'   => [
                [
                    'name'       => '付款',
                    'icon'       => '',
                    'permission' => 'admin.finance.payment.access',
                    'href'       => route('lang.ocadmin.finance.payment.index'),
                    'children'   => [],
                ],
                [
                    'name'       => '退款',
                    'icon'       => '',
                    'permission' => 'admin.finance.refund.access',
                    'href'       => route('lang.ocadmin.finance.refund.index'),
                    'children'   => [],
                ],
            ],
        ];
    }
}
```

**欄位說明：**

| 欄位 | 必填 | 說明 |
|------|------|------|
| `id` | 僅 L1 | HTML id，用於 sidebar 展開/收合狀態 |
| `icon` | 僅 L1 | FontAwesome class |
| `name` | 是 | 顯示名稱 |
| `permission` | 否 | 對應的權限名稱，無此欄位則不做權限檢查 |
| `href` | 是 | 連結，空字串表示純分組節點 |
| `children` | 是 | 子項目陣列 |

### 2.2 MenuComposer 收集與過濾

```php
// app/Portals/Ocadmin/Core/ViewComposers/MenuComposer.php

use App\Portals\Ocadmin\Modules\Catalog\CatalogMenu;
use App\Portals\Ocadmin\Modules\Inventory\InventoryMenu;
use App\Portals\Ocadmin\Modules\Master\MasterMenu;

class MenuComposer
{
    /**
     * 選單註冊表 — 順序即為 sidebar 顯示順序
     */
    protected array $menuClasses = [
        // DashboardMenu::class,
        CatalogMenu::class,
        InventoryMenu::class,
        MasterMenu::class,
        // MemberMenu::class,
        // FinanceMenu::class,
        // SystemMenu::class,
    ];

    public function compose(View $view): void
    {
        $view->with('sys_menus', $this->buildMenus());
    }

    protected function buildMenus(): array
    {
        $user = auth()->user();

        return collect($this->menuClasses)
            ->map(fn ($class) => $class::items())
            ->map(fn ($menu) => $this->filterByPermission($menu, $user))
            ->filter()         // 移除 null（整個群組無權限）
            ->values()
            ->toArray();
    }

    /**
     * 遞迴過濾：移除無權限的項目，子項全空的父層也移除
     */
    protected function filterByPermission(array $item, $user): ?array
    {
        // 檢查項目本身的權限
        if (!empty($item['permission']) && !$user->can($item['permission'])) {
            return null;
        }

        // 遞迴過濾子項目
        if (!empty($item['children'])) {
            $item['children'] = collect($item['children'])
                ->map(fn ($child) => $this->filterByPermission($child, $user))
                ->filter()
                ->values()
                ->toArray();

            // 分組節點（href 為空）且子項全被過濾 → 隱藏父層
            if (empty($item['children']) && empty($item['href'])) {
                return null;
            }
        }

        return $item;
    }
}
```

**過濾邏輯：**

1. 項目有 `permission` 欄位 → 使用者無此權限則移除
2. 項目無 `permission` 欄位 → 不檢查，直接保留
3. 子項目全部被移除且父層 `href` 為空 → 父層也移除（避免空的分組節點）
4. `developer` 的 `can()` 永遠回傳 `true`（見 [0104 §11.5](0104_權限機制.md#十一使用者權限快取)），選單全部可見；`super_admin` 依角色指派的權限決定

**父層可見性規則：** 父層選單（如「系統管理」、「財務會計」）通常不設 `permission`，其可見性由子項向上推導決定：

- 至少一個子項有權限 → 父層**顯示**
- 所有子項都無權限 → 子項全被過濾 → 父層自動**隱藏**

因此不需要為父層選單額外定義選單專用權限（如 `admin.system.menu`），子項的 action 權限（如 `admin.system_access.user.access`）已足夠控制整個選單群組的可見性。

### 2.3 新增模組 Checklist

- [ ] 在模組目錄下建立 `{Module}Menu.php`，實作 `items()` 方法
- [ ] 每個子項目填入對應的 `permission`（通常是 `access` 層級）
- [ ] 在 `MenuComposer::$menuClasses` 中註冊，插入期望的排序位置
- [ ] 確認對應的權限已在 `AclPermissionSeeder` 中定義

## 三、方案 B — DB Driven（獨立選單表）

選單存入獨立的 `sys_menus` / `sys_menu_translations` 表，不與 `acl_permissions` 混用。透過 `permission_name` 欄位關聯權限做可見性判斷。

### 3.1 設計原則

選單與權限各司其職：

| | `sys_menus` 表 | `acl_permissions` 表 |
|---|---|---|
| **職責** | 定義 sidebar 的結構、順序、連結 | 定義功能權限（access / modify / delete） |
| **欄位** | portal、icon、route_name、href、sort_order | name、guard_name |
| **關聯方式** | `menus.permission_name` → `acl_permissions.name` | — |

**不混用的好處：**

- `acl_permissions` 不會多出 `icon`、`href`、`sort_order` 等選單專用欄位
- Spatie 的 `hasPermissionTo()` / `can()` 不會撈到選單節點
- 選單可以有不需權限檢查的純分組節點，不必塞進權限表

### 3.2 資料表結構

```
menus                                  menu_translations
├── id                                 ├── id
├── portal ──→ 'ocadmin','www','pos'   ├── menu_id (FK → menus)
├── parent_id (FK → menus, nullable)   ├── locale
├── permission_name (nullable) ──→     ├── display_name
│   acl_permissions.name               └── unique(menu_id, locale)
├── route_name (nullable)
├── href (nullable)
├── icon (nullable)
├── sort_order (default: 0)
├── is_active (default: true)
└── timestamps
```

**欄位說明：**

| 欄位 | 說明 |
|------|------|
| `portal` | 所屬應用入口：`ocadmin`（後台）、`www`（官網）、`pos`（POS）等，對應 `config/portals.php` |
| `parent_id` | 自我關聯，`null` 表示 L1 根節點 |
| `permission_name` | 關聯到 `acl_permissions.name`，用於可見性判斷。`null` 表示不檢查權限（純分組節點，可見性由子項推導） |
| `route_name` | 本系統的 named route，MenuComposer 用 `route()` 動態產生 URL（含語系前綴） |
| `href` | 外部連結或靜態 URL，直接輸出到前端 |
| `icon` | FontAwesome class，僅 L1 根節點使用 |
| `sort_order` | 同層排序，數字越小越前面 |
| `is_active` | 停用選單項目而不刪除 |

**`route_name` 與 `href` 的優先順序：**

| route_name | href | 結果 |
|---|---|---|
| 有值 | — | `route($route_name)` 產生 URL（本系統路由） |
| null | 有值 | 直接輸出 `href`（外部連結） |
| null | null | 純分組節點（可展開收合的父層） |

### 3.3 樹狀結構範例

以 Ocadmin 後台為例：

```
[portal=ocadmin]

catalog                              (L1, parent_id=null, icon=fa-solid fa-tags)
├── catalog.product                  (L2, permission=admin.catalog.product.access, route=lang.ocadmin.catalog.product.index)
├── catalog.option                   (L2, permission=admin.catalog.option.access, route=lang.ocadmin.catalog.option.index)
└── catalog.category                 (L2, permission=admin.catalog.category.access, route=lang.ocadmin.catalog.category.index)

sale                                 (L1, parent_id=null, icon=fa-solid fa-cart-shopping)
├── sale.order                       (L2, permission=admin.sale.order.access, route=lang.ocadmin.sale.order.index)
└── sale.return                      (L2, permission=admin.sale.return.access, route=lang.ocadmin.sale.return.index)

system                               (L1, parent_id=null, icon=fa-solid fa-gear)
├── system.user                      (L2, permission=admin.system_access.user.access, route=lang.ocadmin.system_access.user.index)
└── system.role                      (L2, permission=admin.system_access.role.access, route=lang.ocadmin.system_access.role.index)
```

> 選單節點只管「呈現與連結」，權限檢查透過 `permission_name` 指向 `acl_permissions` 表。同一個權限可被多個選單節點引用（如 §14 單控制器多選單入口的情境）。

    $table->id();
    $table->foreignId('menu_id')->constrained('sys_menus')->cascadeOnUpdate()->cascadeOnDelete();

### 3.5 Menu Model

```php
// app/Models/Menu.php

namespace App\Models;

use App\Traits\HasTranslation;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Menu extends Model
{
    use HasTranslation;

    protected $fillable = [
        'portal', 'parent_id', 'permission_name',
        'route_name', 'href', 'icon', 'sort_order', 'is_active',
    ];

    protected string $translationModel = MenuTranslation::class;
    protected array $translatedAttributes = ['display_name'];

    protected $with = ['translation'];

    public function parent(): BelongsTo
    {
        return $this->belongsTo(Menu::class, 'parent_id');
    }

    public function children(): HasMany
    {
        return $this->hasMany(Menu::class, 'parent_id')
            ->where('is_active', true)
            ->orderBy('sort_order');
    }

    /**
     * 產生 URL：route_name 優先，其次 href，都沒有回傳空字串
     */
    public function resolveUrl(): string
    {
        if ($this->route_name) {
            return route($this->route_name);
        }

        return $this->href ?? '';
    }

    /**
     * 轉為前端 sidebar 所需的 array 格式
     */
    public function toMenuItem(): array
    {
        return [
            'id'         => 'menu-' . $this->id,
            'icon'       => $this->icon ?? '',
            'name'       => $this->display_name,
            'permission' => $this->permission_name,
            'href'       => $this->resolveUrl(),
            'children'   => $this->children->map(fn ($child) => $child->toMenuItem())->toArray(),
        ];
    }
}
```

### 3.6 MenuComposer（DB 版）

```php
// app/Portals/Ocadmin/Core/ViewComposers/MenuComposer.php

use App\Models\Menu;

class MenuComposer
{
    public function compose(View $view): void
    {
        $view->with('sys_menus', $this->buildMenus());
    }

    protected function buildMenus(): array
    {
        $user = auth()->user();

        // 撈該 portal 的 L1 根節點（含遞迴子項）
        $menus = Menu::with('children.children')  // eager load 2~3 層
            ->where('portal', 'ocadmin')
            ->whereNull('parent_id')
            ->where('is_active', true)
            ->orderBy('sort_order')
            ->get();

        return $menus
            ->map(fn ($menu) => $menu->toMenuItem())
            ->map(fn ($item) => $this->filterByPermission($item, $user))
            ->filter()
            ->values()
            ->toArray();
    }

    /**
     * 遞迴過濾：移除無權限的項目，子項全空的父層也移除
     */
    protected function filterByPermission(array $item, $user): ?array
    {
        if (!empty($item['permission']) && !$user->can($item['permission'])) {
            return null;
        }

        if (!empty($item['children'])) {
            $item['children'] = collect($item['children'])
                ->map(fn ($child) => $this->filterByPermission($child, $user))
                ->filter()
                ->values()
                ->toArray();

            if (empty($item['children']) && empty($item['href'])) {
                return null;
            }
        }

        return $item;
    }
}
```

**父層可見性規則：** 父層選單（如「商品型錄」）的 `permission_name` 設為 `null`，不做權限檢查，可見性由子項向上推導：

```
商品型錄  (permission_name = null, 純分組節點)
├── 商品      (permission_name = admin.catalog.product.access)
├── 選項      (permission_name = admin.catalog.option.access)
└── 規格      (permission_name = admin.catalog.specification.access)
```

- 使用者有任一子項權限 → 「商品型錄」**顯示**
- 使用者三個都沒有 → 子項全被過濾 → 「商品型錄」自動**隱藏**

因此不需要為父層選單額外定義權限（如 `admin.catalog.menu`），子項的 action 權限（如 `admin.catalog.product.access`）已足夠控制整個選單群組的可見性。`developer` 的 `can()` 永遠回傳 `true`（見 [0104 §10.5](0104_權限機制.md#十使用者權限快取)），選單全部可見；`super_admin` 依角色指派的權限決定。

### 3.7 Seeder 範例

```php
// database/seeders/MenuSeeder.php

use App\Models\Menu;

class MenuSeeder extends Seeder
{
    public function run(): void
    {
        $portal = 'ocadmin';

        // L1: 商品型錄
        $catalog = Menu::updateOrCreate(
            ['portal' => $portal, 'route_name' => null, 'permission_name' => null, 'parent_id' => null, 'icon' => 'fa-solid fa-tags'],
            ['sort_order' => 10, 'is_active' => true]
        );
        $catalog->translateOrNew('en')->display_name = 'Catalog';
        $catalog->translateOrNew('zh_Hant')->display_name = '商品型錄';
        $catalog->save();

        // L2: 商品
        $product = Menu::updateOrCreate(
            ['portal' => $portal, 'parent_id' => $catalog->id, 'permission_name' => 'admin.catalog.product.access'],
            ['route_name' => 'lang.ocadmin.catalog.product.index', 'sort_order' => 1]
        );
        $product->translateOrNew('en')->display_name = 'Products';
        $product->translateOrNew('zh_Hant')->display_name = '商品';
        $product->save();

        // L2: 選項
        $option = Menu::updateOrCreate(
            ['portal' => $portal, 'parent_id' => $catalog->id, 'permission_name' => 'admin.catalog.option.access'],
            ['route_name' => 'lang.ocadmin.catalog.option.index', 'sort_order' => 2]
        );
        $option->translateOrNew('en')->display_name = 'Options';
        $option->translateOrNew('zh_Hant')->display_name = '選項';
        $option->save();

        // ... 其餘模組依此類推
    }
}
```

### 3.8 新增模組 Checklist

- [ ] 在 `MenuSeeder` 中新增該模組的 L1 + L2 選單項目
- [ ] 填入 `permission_name`（通常指向 `*.access` 權限）
- [ ] 填入 `route_name`（本系統路由）或 `href`（外部連結）
- [ ] 確認對應的權限已在 `AclPermissionSeeder` 中定義
- [ ] 執行 `php artisan db:seed --class=MenuSeeder`

### 3.9 多 Portal 支援

同一張 `sys_menus` 表透過 `portal` 欄位區分不同應用的選單：

| portal | 應用 | 說明 |
|--------|------|------|
| `ocadmin` | 後台管理 | 完整的 sidebar 選單樹 |
| `www` | 官網 ApiWww | 前台導覽列 |
| `pos` | POS 系統 ApiPos | POS 功能選單 |

每個 Portal 的 MenuComposer 只撈自己的 `portal`：

```php
// Ocadmin
Menu::where('portal', 'ocadmin')->whereNull('parent_id')...

// POS
Menu::where('portal', 'pos')->whereNull('parent_id')...
```

不同 Portal 的選單完全獨立，不會互相干擾。同一個權限（如 `admin.catalog.product.access`）可被多個 Portal 的選單節點引用。

## 四、方案評估比較

### 4.1 比較

| | A. Code Driven | B. DB Driven（獨立選單表） |
|---|---|---|
| **選單定義位置** | PHP 檔案（版控追蹤） | `sys_menus` 表 |
| **修改方式** | 改 code + deploy | 改 DB（後台 UI 或 seeder） |
| **可讀性 / debug** | 打開檔案就看到 | 要查 DB |
| **DB query** | 0（純 PHP array） | 1 次（可快取） |
| **Seeder 維護** | 不需要 | 需維護 parent_id 對應關係 |
| **後台 UI 管理選單** | 不支援 | 支援 |
| **新增模組** | 寫 Menu class + 註冊 | 寫 seeder + 跑 seed |
| **多語支援** | 翻譯檔（lang/*.php） | `sys_menu_translations` 翻譯表，新增語系只需加記錄 |
| **多 Portal** | 每個 Portal 各自維護 MenuComposer + Menu class | 同一張表，`portal` 欄位區分 |
| **角色組合快取** | 每次請求都遍歷 Menu array + 權限過濾 | 可搭配角色組合快取，同角色組合的使用者共享快取結果 |
| **大型系統（數百～數千支作業）** | Menu class 散落數十個檔案，難以全局掌握 | 一張表統一管理，可搜尋、排序、批次調整 |
| **權限與選單一致性** | 需手動確保 Menu class 的 permission 欄位與 DB 權限名稱吻合 | `permission_name` 明確指向 `acl_permissions.name` |
| **權限表汙染** | 無 | 無（選單獨立表，不混入 `acl_permissions`） |
| **適合場景** | 選單少、結構穩定的初期階段 | 系統規模較大，或需要後台 UI 管理選單、多 Portal 支援 |

### 4.2 效能比較：每次請求的成本

方案 A（Code driven）每次請求都要：

1. 載入完整選單定義（不管單檔多大、或是拆多少模組選單檔案，都要全部載入）
2. 遍歷每個節點呼叫 `$user->can()` 做權限過濾
3. 產生最終選單傳給 View

PHP 檔案本身的載入很快（OPcache 會編譯快取 bytecode），瓶頸不在讀檔，而在**步驟 2 的權限過濾**。

假設 2000 支作業、3 層選單大約 2500 個節點：

| 做法 | 每次請求的成本 |
|---|---|
| **Code driven** | 遍歷 2500 節點 × 每個呼叫 `$user->can()` = 2500 次權限判斷 |
| **DB driven + 角色組合快取** | 第一次：1 次 DB query + 遍歷過濾 → 存快取。之後：1 次 cache hit，0 次權限判斷 |

Code driven 當然也可以加快取，但那就等於自己重新實作一遍——以角色組合為 key 快取過濾結果。做到最後，跟 DB driven + 快取殊途同歸，只是選單來源從 DB 換成檔案而已。

系統規模小的時候感受不到差異，規模大了之後 DB driven + 快取的優勢就很明顯。

### 4.3 結論

**預計採用方案 B（DB 驅動，獨立選單表）。** 選單與權限分離，各司其職：`sys_menus` 表管呈現與導航，`acl_permissions` 表管功能權限。搭配 `portal` 欄位支援多應用入口，搭配角色組合快取與翻譯表，在效能、多語、維護便利性上都更具優勢。

> **備註：** 不論方案 A 或 B，選單維護都是開發人員的工作——選單必須有對應的程式（Controller、Route）才能運作。兩者的差別在於維護方式：方案 A 改檔案後需重新部署；方案 B 透過 DB 操作即時生效，調整選單順序、顯示名稱、啟用/停用時不需部署，開發與測試階段更為便利。
