# 集團／公司／企業與部門的資料庫設計

## 1. 設計原則

本系統將組織實體分為**外部**與**內部**兩類，分別使用不同資料表管理：

| 分類 | 資料表 | 用途 |
|------|--------|------|
| 外部 | `organizations` | 客戶、廠商、經銷商等外部公司 |
| 內部 | `companies` + `departments` | 自家公司與其部門結構 |

此設計適用於所有系統（人資、電商等），不限特定領域。

### 1.1 為什麼這樣分

- **語意明確**：`companies`（公司）搭配 `departments`（部門）在語意上天然配對，表達「公司底下有部門」的階層關係直覺清晰。
- **外部公司不需記錄部門**：外部往來對象（客戶、廠商、經銷商）通常只需公司層級資料，不需追蹤其內部部門結構。
- **職責分離**：外部與內部分開管理，避免在同一張表中混雜不同性質的資料。

## 2. 資料表結構

### 2.1 organizations（外部公司）

```
organizations
├── id
├── parent_id          -- 自引用階層（集團→子公司）
├── business_no        -- 統一編號
├── shipping_state     -- 州/省/縣市
├── shipping_city      -- 區/鄉/鎮
├── shipping_address1  -- 地址1
├── shipping_address2  -- 地址2
└── translations (name, short_name)
```

- 統一管理所有外部往來對象：**客戶、廠商、經銷商**等
- 不區分客戶或廠商的表，一律使用 `organizations`，透過標籤或關聯區分角色
- 支援 `parent_id` 自引用，可表達外部集團的母子公司關係
- 具備多語系翻譯、地址等基本資料
- **不需要** departments 子結構

### 2.2 companies（內部公司）

```
companies
├── id
├── parent_id          -- 自引用階層（集團→子公司）
├── code               -- 公司代碼（唯一）
├── business_no        -- 統一編號
├── phone
├── address
├── is_active
├── sort_order
└── translations (name, short_name)
```

- 代表自家營運的法人實體
- 支援 `parent_id` 自引用，可建立集團化階層結構
- 具備多語系翻譯
- 透過 `company_user` 中介表關聯使用者

### 2.3 departments（部門）

```
departments
├── id
├── company_id         -- 歸屬公司
├── parent_id          -- 自引用階層（處→部→課→組）
├── name
├── code
├── is_active
└── sort_order
```

- 隸屬於特定的內部公司（`company_id`）
- 支援多層級部門結構（處→部→課→組）
- 僅用於內部公司，外部公司不使用

## 3. 集團化階層管理

`organizations` 與 `companies` 都透過 `parent_id` 自引用來表達集團階層：

### 3.1 內部公司的集團結構

```
companies (parent_id = null)  → 集團母公司 / 控股公司
├── companies (parent_id = 1) → 子公司 A
├── companies (parent_id = 1) → 子公司 B
└── companies (parent_id = 1) → 子公司 C
```

### 3.2 外部公司的集團結構

```
organizations (parent_id = null)  → 外部集團總公司
├── organizations (parent_id = 1) → 該集團子公司 A
└── organizations (parent_id = 1) → 該集團子公司 B
```

不需要額外建立 `groups` 或 `holdings` 表，`parent_id` 自引用已足夠。

## 4. 為什麼不用 teams 取代 departments

### 4.1 公司與部門具有本質差異

`companies` 與 `departments` 在業務上有明確的職責劃分：

- **公司**：法人實體，涉及稅務、勞保、發薪、財務報表等
- **部門**：行政單位，用於人員歸屬、管理層級、簽核流程等

若用 `teams` 統一表示，會需要 `if ($team->type === 'company')` 這類隱性耦合判斷，模糊了兩者的本質差異。

### 4.2 teams 的適用場景

若未來有需求，`teams` 可作為**工作分組**使用，但不取代制度性的公司與部門結構：

- 專案小組
- 跨部門任務編組
- 臨時活動人員

## 5. 完整組織階層圖

```
organizations（外部公司）
├── [parent_id 自引用：外部集團 → 外部子公司]
└── 客戶、廠商、經銷商等往來對象

companies（內部公司）
├── [parent_id 自引用：集團 → 子公司]
└── departments（部門）
    └── [parent_id 自引用：處 → 部 → 課 → 組]
```

## 6. 總結

| 決策 | 內容 |
|------|------|
| `organizations` | 所有外部公司（客戶、廠商、經銷商），支援 parent_id 集團階層，不需部門子結構 |
| `companies` | 內部公司（自家法人實體），支援 parent_id 集團階層 |
| `departments` | 內部公司底下的行政單位，支援 parent_id 多層級部門 |
| `teams` | 不用於制度性功能；若有工作分組需求可再評估 |
