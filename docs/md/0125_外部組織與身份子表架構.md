# 外部組織與身份子表架構

## 1. 概述

系統中與公司往來的外部對象（供應商、客戶、經銷商等）統一以 `org_organizations` 作為主表管理。各身份的專屬欄位，透過**身份子表**擴充，而非獨立建表。

### 1.1 與內部公司的區別

| | 內部公司 | 外部組織 |
|---|---|---|
| 用途 | 集團旗下企業、部門 | 與公司往來的外部對象 |
| 表名 | `org_companies` / `hrm_companies` | `org_organizations` |
| 關聯 | 員工歸屬、帳號存取權 | 訂單、採購、商品供應 |
| 參考文件 | [0307_多公司與帳號設計](0307_多公司與帳號設計.md) | 本文件 |

## 2. 主表 `org_organizations`

存放所有外部組織的**共用欄位**——不論身份為何都會用到的資料：

| 欄位 | 說明 |
|------|------|
| name / short_name | 名稱（透過 translations 子表支援多語） |
| tax_id_number / business_no | 統一編號 |
| telephone, fax | 聯絡電話 |
| contact_name, contact_phone, contact_email | 聯絡人 |
| country_code | 國家代碼 |
| shipping_state, shipping_city, shipping_address | 地址 |

> 各分支的欄位略有不同，依業務需求調整。上表為常見欄位的聯集。

## 3. 身份子表模式（Identity Subtable Pattern）

### 3.1 設計理念

一間外部公司可能同時具備多種身份——既是供應商又是客戶。若為每種身份建獨立主表，同一間公司的基本資料（名稱、統編、地址）會重複儲存，且難以關聯。

做法是：**主表存共用資料，子表存身份專屬資料。有子表記錄即代表具備該身份。**

```
org_organizations (主表：所有外部公司)
  ├── org_organization_suppliers   (供應商專屬欄位)
  ├── org_organization_customers   (客戶專屬欄位)
  └── org_organization_{identity}  (未來其他身份)
```

### 3.2 子表範例：`org_organization_suppliers`

| 欄位 | 類型 | 說明 |
|------|------|------|
| id | bigint | PK |
| organization_id | bigint FK | → org_organizations（unique，1:1） |
| default_currency_code | varchar(3) | 預設交易幣別 |
| payment_term | varchar | 預設付款條件 |
| is_factory | boolean | 是否為代工工廠 |
| lead_time_days | int | 預設前置天數 |
| notes | text | 備註 |

### 3.3 子表範例：`org_organization_customers`（尚未實作，示意）

| 欄位 | 類型 | 說明 |
|------|------|------|
| id | bigint | PK |
| organization_id | bigint FK | → org_organizations（unique，1:1） |
| credit_limit | decimal | 信用額度 |
| payment_term | varchar | 預設付款條件 |
| sales_rep_user_id | bigint FK | 負責業務 |
| notes | text | 備註 |

### 3.4 判斷身份

不需要額外的 type 欄位或多對多 pivot，直接查子表：

```php
// 判斷是否為供應商
$org->supplier !== null

// 取得所有供應商
Organization::has('supplier')->get()

// 取得既是供應商又是客戶的組織
Organization::has('supplier')->has('customer')->get()
```

## 4. Model 關聯

```php
// Organization.php
public function supplier(): HasOne
{
    return $this->hasOne(OrganizationSupplier::class);
}

public function customer(): HasOne
{
    return $this->hasOne(OrganizationCustomer::class);
}

// OrganizationSupplier.php
public function organization(): BelongsTo
{
    return $this->belongsTo(Organization::class);
}
```

## 5. 其他模組的引用方式

其他模組（如商品、訂單）引用外部組織時，FK 統一指向 `org_organizations.id`：

```php
// ctl_product_suppliers（商品的供應商報價）
$table->foreignId('organization_id')->constrained('org_organizations');

// sal_orders（訂單的客戶/供應商）
$table->foreignId('organization_id')->nullable()->constrained('org_organizations');
```

語意由上下文決定——在 `ctl_product_suppliers` 裡的 organization 就是供應商，在銷售訂單裡的 organization 就是客戶。

## 6. 與 OrganizationIdentity Enum 的關係

branchHuabing 與 branchSunline 都定義了 `OrganizationIdentity` enum：

```php
enum OrganizationIdentity: string
{
    case Dealer = 'dealer';
    case Customer = 'customer';
    case Supplier = 'supplier';
}
```

此 enum 用於**程式邏輯中的身份識別**（如 UI 篩選、表單選項），不直接儲存在 organizations 表。身份的持久化由子表負責。

## 7. 何時該拆子表、何時不用

| 情境 | 建議 |
|------|------|
| 該身份有 3+ 個專屬欄位 | 建子表 |
| 只需要知道「是不是這個身份」，無專屬欄位 | 不建子表，用 enum 或 pivot 即可 |
| 專屬欄位只有 1-2 個且不太可能增加 | 可先放主表，nullable |

## 8. 設計優勢

- **不重複**：一間公司只有一筆主表記錄，名稱、統編、地址不重複
- **可擴展**：新增身份只需加子表，不改主表 schema
- **多重身份**：一間公司可同時具備多種身份，互不衝突
- **查詢直覺**：`Organization::has('supplier')` 語意清楚
- **簡單**：本質上是 1:1 延伸表，不是複雜的多型設計
