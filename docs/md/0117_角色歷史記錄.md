# 角色歷史與系統使用者追蹤

> 建立日期：2026-02-26

---

## 一、問題背景

### 1.1 現況缺失

LaravelOcadmin 使用 Spatie Laravel Permission 管理角色，但角色移除時 `acl_model_has_roles` 的記錄直接被刪除，**無法回答以下問題**：

| 問題 | 現況 |
|------|------|
| 張三什麼時候被授予後台權限的？ | 無法查詢 |
| 上個月有哪些人被移除了後台權限？ | 無法查詢 |
| 這個管理帳號多久沒登入了？ | 無法查詢 |
| 離職員工的權限回收是否確實執行？ | 只能間接判斷「目前沒有角色」 |

### 1.2 解決方案

引入 `acl_system_users` 資料表，以「存取期間追蹤」記錄角色授予與移除的歷史。角色移除時標記 `revoked_at`，不刪除記錄；重新授予時建立新記錄，完整保留每段授權期間。

## 二、`app` 欄位設計

### 2.1 `app` 的含意

`app` 欄位代表「這筆記錄追蹤的是哪一組角色」，值為穩定的系統級識別碼：

```
app = 'super_admin'  →  super_admin 角色
app = 'admin'        →  admin.* 角色群組
app = 'ess'          →  （未來）ess.* 角色群組
```

`app` 不是 Portal 目錄名稱。一個 `app` 值可能對應多個 Portal。

### 2.2 多 Portal 共用同一 `app`

小型系統中，為每個 Portal 建立獨立角色前綴會過度設計。多個 Portal 共用同一組 `admin.*` 角色是合理的簡化：

```
app\Portals\Ocadmin ─┐
app\Portals\Api     ─┤── 共用 admin.* 角色 ── app = 'admin'
app\Portals\Report  ─┘

app\Portals\App     ─── 使用 ess.* 角色  ── app = 'ess'（未來需要時再追蹤）
```

### 2.3 為什麼不叫 `portal`

1. **一對多關係**：一個 `app` 值對應多個 Portal 目錄，用 `portal` 容易誤解為一對一
2. **可自訂應用**：可以用 Portal 名稱，但小系統可能多個 Portal 全部使用 `admin.*` 前綴，欄位命名為 `app` 讓值的定義更彈性

## 三、資料表結構

### 3.1 acl_system_users

```sql
CREATE TABLE acl_system_users (
    id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id         BIGINT UNSIGNED,               -- users.id
    app             VARCHAR(20) NOT NULL,           -- 角色群組識別（super_admin, admin, ess, ...）
    enrolled_at     TIMESTAMP NULL,                 -- 授予角色時間
    revoked_at      TIMESTAMP NULL,                 -- 移除角色時間
    last_login_at   TIMESTAMP NULL,                 -- 最後登入時間
    created_at      TIMESTAMP NULL,
    updated_at      TIMESTAMP NULL,

    INDEX (user_id, app),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

**設計要點**：

- 自增 `id` 作為 PK，每段授權期間獨立一筆記錄
- 同一使用者可同時有 `super_admin` 和 `admin` 兩筆 active 記錄，互不干擾
- 完整保留歷史：revoke 後再授予會產生新記錄，不覆寫舊記錄

**資料範例**：

```
id | user_id | app         | enrolled_at | revoked_at  | 說明
1  | 1       | admin       | 2026-01     | 2026-03     | 第一段 admin 期間（已結束）
2  | 1       | super_admin | 2026-06     | NULL        | 目前是 super_admin
3  | 1       | admin       | 2026-06     | NULL        | 目前也有 admin 角色
4  | 2       | admin       | 2026-02     | NULL        | user 2 的 admin 期間
```

### 3.2 時間欄位分類

| 分類 | 欄位 | 觸發時機 | 用途 |
|------|------|---------|------|
| 角色生命週期 | `enrolled_at` | UserController 授予角色時 | 記錄加入時間 |
| 角色生命週期 | `revoked_at` | UserController 移除角色時 | 記錄移除時間 |
| 登入事件 | `last_login_at` | 登入成功時（更新所有 active 記錄） | 帳號安全審計 |

## 四、Portal 角色檢查

### 4.1 RequirePortalRole Middleware

原本 Ocadmin 使用 `AccessBackend` middleware，硬編碼檢查 `admin.*`。改為參數化的 `RequirePortalRole`，可複用於不同 Portal：

```php
class RequirePortalRole
{
    public function handle(Request $request, Closure $next, string $prefix): Response
    {
        $user = $request->user();

        if (!$user || !$user->hasPortalRole($prefix)) {
            if ($request->expectsJson()) {
                abort(401, '未授權');
            }
            return redirect()->guest('/admin/login');
        }

        return $next($request);
    }
}
```

### 4.2 User Model 方法

```php
public function hasPortalRole(string $prefix): bool
{
    return $this->roles->contains(
        fn ($role) => $role->name === 'super_admin'
            || str_starts_with($role->name, $prefix . '.')
    );
}

// hasBackendRole() 保留為便捷別名
public function hasBackendRole(): bool
{
    return $this->hasPortalRole('admin');
}
```

### 4.3 Middleware 掛載

```php
// bootstrap/app.php
$middleware->alias([
    'requirePortalRole' => \App\Http\Middleware\RequirePortalRole::class,
]);

// Ocadmin 路由
Route::middleware(['auth', 'requirePortalRole:admin'])->group(...);

// 未來 API Portal（共用 admin.* 角色）
Route::middleware(['auth', 'requirePortalRole:admin'])->group(...);

// 未來 App Portal（ESS）
Route::middleware(['auth', 'requirePortalRole:ess'])->group(...);
```

## 五、角色連動規則

當 UserController 儲存使用者角色時，呼叫 `SystemUser::syncFromRoles($user)` 同步更新。

**核心邏輯**：`super_admin` 和 `admin.*` 各自獨立追蹤，互不干擾。

### 5.1 各 app 群組的判斷

| app 群組 | 判斷條件 | 說明 |
|---------|---------|------|
| `super_admin` | `$user->hasRole('super_admin')` | 獨立追蹤 |
| `admin` | 有任一 `admin.*` 角色 | 獨立追蹤 |

### 5.2 連動動作

對每個 app 群組獨立執行：

| 情境 | 動作 |
|------|------|
| 有角色，無 active 記錄 | 新增記錄（`enrolled_at = now()`） |
| 無角色，有 active 記錄 | 關閉記錄（`revoked_at = now()`） |
| 有角色，有 active 記錄 | 不動 |
| 無角色，無 active 記錄 | 不動 |

### 5.3 程式碼

```php
public static function syncFromRoles(User $user, string $prefix = 'admin'): void
{
    // super_admin 獨立追蹤
    static::syncApp($user, 'super_admin', $user->hasRole('super_admin'));

    // prefix 角色獨立追蹤
    $hasPrefixRoles = $user->roles->contains(
        fn ($role) => str_starts_with($role->name, $prefix . '.')
    );
    static::syncApp($user, $prefix, $hasPrefixRoles);
}

protected static function syncApp(User $user, string $app, bool $hasRoles): void
{
    $active = static::where('user_id', $user->id)
        ->where('app', $app)
        ->whereNull('revoked_at')
        ->first();

    if ($hasRoles && !$active) {
        static::create([
            'user_id' => $user->id,
            'app' => $app,
            'enrolled_at' => now(),
        ]);
    } elseif (!$hasRoles && $active) {
        $active->update(['revoked_at' => now()]);
    }
}
```

## 六、追蹤範圍

### 6.1 初期追蹤範圍

| 角色 | `app` 值 | 說明 |
|------|---------|------|
| `super_admin` | `'super_admin'` | 最高權限，需稽核 |
| `admin.*` | `'admin'` | 後台管理權限，需稽核 |
| `ess.*` | `'ess'` | 員工自助服務權限，需稽核 |

### 6.2 常用查詢

```sql
-- 目前有效的所有管理者
SELECT * FROM acl_system_users WHERE revoked_at IS NULL;

-- 某使用者的完整授權歷史
SELECT * FROM acl_system_users WHERE user_id = ? ORDER BY enrolled_at;

-- 上個月被移除權限的人
SELECT * FROM acl_system_users
WHERE revoked_at BETWEEN '2026-01-01' AND '2026-01-31';

-- 超過 90 天未登入的 active 管理者
SELECT * FROM acl_system_users
WHERE revoked_at IS NULL AND last_login_at < NOW() - INTERVAL 90 DAY;
```

## 七、權限檢查流程

整合後的完整流程：

```
1. 使用者登入
   └── 更新該使用者所有 active 記錄的 last_login_at

2. 進入 Portal
   └── RequirePortalRole:admin middleware
       ├── super_admin → 放行
       ├── 有任一 admin.* 角色 → 放行
       └── 其他 → redirect 或 401

3. 操作功能
   └── Spatie Permission 個別權限檢查 can('...')
```
