# 原始打卡表

## 1. 設計目標

儲存所有打卡的原始資料，作為差勤系統的基礎資料來源。每次打卡產生一筆記錄，後續的每日統計與每月統計皆基於這些原始記錄計算。

### 1.1 為什麼需要原始打卡表

1. **保留完整歷史**：員工一天可能多次進出，需完整記錄
2. **資料來源追溯**：可追蹤每筆記錄的來源（設備、網頁、APP、手動）
3. **異常處理**：統計異常時可回溯原始記錄查明原因
4. **審計需求**：符合勞動法規對出勤記錄的保存要求

### 1.2 與每日統計的關係

```
原始打卡表（clock_records）── 一天多筆
    08:55  in （門禁進門）
    12:00  out（門禁出門）
    13:05  in （門禁進門）
    18:10  out（門禁出門）
              ↓ 每日批次處理
每日出勤統計（daily_attendances）── 一天一筆
    上班：08:55（第一個 in）
    下班：18:10（最後一個 out）
    工時：8 小時 15 分
```

## 2. 資料表設計

### clock_records

```
clock_records
├── id
├── employee_id          -- FK → employees
├── clocked_at           -- 打卡時間（dateTime）
├── clock_type           -- 打卡類型：in / out / other（見 §3）
├── clock_method         -- 打卡方式：device / web / app / manual / import（見 §4）
│
├── device_id            -- 設備編號（nullable, varchar 50）
├── device_name          -- 設備名稱（nullable, varchar 100）
├── location             -- 打卡地點（nullable, varchar 100）
│
├── latitude             -- 緯度（nullable, decimal 10,7）
├── longitude            -- 經度（nullable, decimal 10,7）
│
├── verification_method  -- 驗證方式：face / fingerprint / card / password（nullable）
├── verification_score   -- 辨識信心分數 0-100（nullable, decimal 5,2）
├── photo_path           -- 打卡照片路徑（nullable）
│
├── status               -- 審核狀態：pending / approved / rejected / voided（見 §5）
├── note                 -- 備註（nullable, text）
├── approved_by          -- 審核人 ID（nullable, FK → users）
├── approved_at          -- 審核時間（nullable）
│
├── ip_address           -- 打卡 IP（nullable, varchar 45）
├── user_agent           -- 瀏覽器/APP 資訊（nullable, text）
│
├── timestamps
└── soft_deletes

INDEX(employee_id, clocked_at)
INDEX(clocked_at)
INDEX(status)
INDEX(clock_method)
UNIQUE(employee_id, clocked_at, clock_method)
```

## 3. 打卡類型（clock_type）

| 值 | 說明 | 使用場景 |
|----|------|----------|
| `in` | 進入 | 上班打卡、門禁進門、外出後返回 |
| `out` | 離開 | 下班打卡、門禁出門、外出用餐 |
| `other` | 其他 | 無法歸類的特殊情況（保留欄位） |

**設計理念**：
- 語意放寬：`in/out` 不僅是「上班/下班」，而是廣義的「進/出」
- 智能判斷：系統自動判斷第一個 `in` = 上班，最後一個 `out` = 下班
- 彈性應用：適用於傳統打卡機、門禁系統、APP 打卡等各種場景

## 4. 打卡方式（clock_method）

| 值 | 說明 | 特點 |
|----|------|------|
| `device` | 實體設備 | 打卡鐘、指紋機、人臉辨識機、門禁讀卡機 |
| `web` | 網頁打卡 | 適合居家辦公或辦公室電腦打卡 |
| `app` | 行動 APP | 支援 GPS 定位，適合外勤人員 |
| `manual` | 手動補登 | 忘記打卡時由主管或 HR 補登，預設 status = pending |
| `import` | 批次匯入 | 從外部系統或 CSV 檔案匯入 |

## 5. 審核狀態（status）

| 值 | 說明 | 觸發情境 |
|----|------|----------|
| `approved` | 已審核通過 | 設備/網頁/APP 打卡預設直接通過 |
| `pending` | 待審核 | 手動補登時自動設定 |
| `rejected` | 已駁回 | 審核人員駁回補登申請 |
| `voided` | 已作廢 | 誤打卡或重複記錄作廢 |

## 6. 應用場景

### 場景 1：傳統打卡機

```
08:55 按「上班」→ clock_type=in,  clock_method=device
18:10 按「下班」→ clock_type=out, clock_method=device
```

### 場景 2：門禁系統（自動記錄）

```
08:55 刷卡進門 → clock_type=in,  clock_method=device, device_id=GATE-01
12:00 刷卡出門 → clock_type=out, clock_method=device
13:05 刷卡進門 → clock_type=in,  clock_method=device
18:10 刷卡出門 → clock_type=out, clock_method=device

處理結果：
  上班時間：08:55（第一個 in）
  下班時間：18:10（最後一個 out）
```

### 場景 3：APP 打卡 + GPS

```
clock_method=app
latitude=25.0330, longitude=121.5654
→ 系統驗證是否在允許範圍內（見 §8）
```

### 場景 4：手動補登

```
員工忘記打卡 → HR 補登
  clock_method=manual, status=pending
  → 主管審核 → approved / rejected
```

## 7. 核心業務邏輯

### 7.1 防重複打卡

同一員工在 **5 分鐘內**相同 `clock_type` 的記錄視為重複，系統拒絕新增。

```
驗證邏輯：
  WHERE employee_id = ?
  AND   clock_type = ?
  AND   clocked_at BETWEEN (目標時間 - 5min) AND (目標時間 + 5min)
  → 有記錄 → 拒絕（拋出 DuplicateClockException）
```

### 7.2 唯一約束

資料庫層級：`UNIQUE(employee_id, clocked_at, clock_method)`，確保同一員工、同一時間、同一方式不會重複。

### 7.3 自動設定審核狀態

| clock_method | 預設 status |
|-------------|-------------|
| device / web / app / import | `approved` |
| manual | `pending`（需審核） |

## 8. GPS 位置驗證

APP 打卡時，系統驗證打卡地點是否在允許範圍內：

```
config/hrm.php → clock_locations
  [
    { name: '台北總部',   lat: 25.0330, lng: 121.5654, radius: 100m },
    { name: '新竹分公司', lat: 24.8138, lng: 120.9675, radius: 100m },
  ]

驗證：Haversine 公式計算距離 ≤ radius → 通過
```

## 9. 匯入匯出

### 9.1 CSV 格式

```csv
employee_number,clocked_at,clock_type,device_id,location
E001,2025-10-10 08:55:30,in,GATE-01,台北總部大門
E001,2025-10-10 18:10:45,out,GATE-01,台北總部大門
E002,2025-10-10 09:02:15,in,GATE-01,台北總部大門
```

### 9.2 匯入流程

1. 上傳 CSV 或透過 artisan command `clock:import {file}`
2. 依 `employee_number` 比對員工
3. 以 `clock_method=import` 建立記錄
4. 回報匯入結果

### 9.3 匯出流程

依日期範圍匯出 CSV，欄位：employee_number, employee_name, clocked_at, clock_type, device_id。

## 10. 與其他表的關聯

### 10.1 資料流向

```
clock_records（原始打卡）
    ↓ 每日批次處理
daily_attendances（每日出勤統計）
    ↓ 每月彙總
monthly_attendances（每月出勤統計）
```

### 10.2 每日統計取值

```
當天所有 approved 記錄，依 clocked_at 排序：
  clocked_in  ← 第一筆 clock_type=in
  clocked_out ← 最後一筆 clock_type=out
```

### 10.3 與行事曆的關係

每日統計需結合行事曆（見 [0301_行事曆管理](0301_行事曆管理.md)）判斷：

| 判斷 | 依據 |
|------|------|
| 今天是否為工作日 | 行事曆 day_type |
| 是否遲到／早退 | 工作日 + 班別時間 |
| 是否算加班 | 休假日出勤 or 工作日超時 |

## 11. 資料保留與效能

### 11.1 資料分區（大量資料時）

建議按月份分區（MySQL 8.0+ RANGE PARTITION），以 `clocked_at` 的年月為分區鍵。

### 11.2 資料保留政策

依勞動法規保存至少 **5 年**（建議 7 年）。過期資料先匯出備份 CSV 後再刪除。
