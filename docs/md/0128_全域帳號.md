# 全域帳號

## 1. 設計目標

在 `users` 表中預先建立一組**系統層級的內建帳號**，作為所有自動操作的掛名身份與管理入口。這些帳號不是業務角色（店長、會計、人資等），而是**平台初始化就存在、永遠不會被刪除**的全域帳號。

### 1.1 為什麼需要

| 情境 | 若沒有全域帳號 | 有全域帳號 |
|------|---------------|------------|
| Seeder / Migration 補資料 | `created_by = null` 或亂塞 | 掛 `system` |
| Queue / 排程任務 | `updated_by = null` | 掛 `system` |
| 緊急除錯 / 資料修復 | 找個真人帳號代勞 | 有 `admin` 可用 |
| 展示 / 教學 | 用真實員工帳號（有洩漏風險） | 有 `demo` |
| 老闆只看報表 | 給太大權限 | 有 `reader` |

## 2. 推薦的全域帳號（第一版）

**建立五個全域帳號：**

| username | 可登入 | 用途 | company 範圍 | 可刪除 |
|----------|--------|------|--------------|--------|
| `system`  | ❌ | 平台底層 fallback（Framework / Seeder / 不知道是誰） | 跨所有公司 | ❌ |
| `service` | ❌ | 應用層自動化流程（webhook / API 介接 / 排程同步） | 跨所有公司 | ❌ |
| `admin`   | ✅ | 最高管理者（平台維護者） | 跨所有公司 | ❌ |
| `demo`    | ✅ | 展示 / 教學 / UI 驗收 | 綁測試公司 | ❌ |
| `reader`  | ✅ | 唯讀（報表、稽核、查資料） | 依需求指定 | ❌ |

### 2.1 `system` vs `service` 的分工

兩者都是「不給人登入、只作掛名」的機器身份，但**語意層級不同**：

| 面向 | `system` | `service` |
|------|----------|-----------|
| 層級 | **平台底層 / Framework** | **應用層 / 業務流程** |
| 觸發來源 | 不知道是誰做的、沒有明確業務 context | 某個明確的自動化流程代替人做 |
| 典型情境 | Seeder 初始化、Migration 補資料、Queue worker 無 actor、Framework 內部修復 | webhook 回寫、排程同步、API 介接、批次匯入、外部平台資料更新 |
| 一句話定義 | **「平台自己做的」** | **「某個服務流程代替人做的」** |
| Log 可讀性 | 「這筆來源不明，走系統 fallback」 | 「這筆是 XX 流程自動建立的」 |

**判斷口訣**：寫程式時問自己「這個動作有沒有明確的業務意圖？」
- 有 → `service`（例如金流 webhook 進來更新訂單）
- 沒有 → `system`（例如 seeder 補一筆預設值）

> 第一版即使只有一個 `service`，語意上已經與 `system` 有差別。未來當業務介接變多，可再細分 `service_web` / `service_hrm_sync` / `service_invoice`（見 [第 7 節](#7-未來演進service-的細分)）。

### 2.2 為什麼不建 `tester` / `developer` / `guest`

- `tester`：與 `demo` 重疊，除非有正式 QA 流程，否則先合併
- `developer`：偏技術維護，初期與 `admin` 角色幾乎相同，等需要拆權限時再加
- `guest`：未登入本來就是 guest，不需要對應一筆 user

## 3. 資料層設計

### 3.1 users 表新增欄位

```php
Schema::table('users', function (Blueprint $table) {
    $table->boolean('is_system')->default(false)->after('id')->comment('系統內建帳號，不可刪除/改名');
    $table->boolean('is_loginable')->default(true)->after('is_system')->comment('是否允許登入');
});
```

| 欄位 | 類型 | 說明 |
|------|------|------|
| `is_system` | boolean | 標記為系統內建帳號，Observer 會阻擋刪除與關鍵欄位變更 |
| `is_loginable` | boolean | 是否允許登入。`system` 帳號設 `false`，只能作為 actor 掛名 |

> 若已有 `user_type` 或 `type` enum 欄位，可改用 `type = 'system'` 取代 `is_system`，擇一即可。

### 3.2 固定 ID

Seeder 以明確指定 ID 寫入，避免被 auto-increment 占用：

| username  | id |
|-----------|----|
| `system`  | 1 |
| `service` | 2 |
| `admin`   | 3 |
| `demo`    | 4 |
| `reader`  | 5 |

> 寫入後需 `ALTER TABLE users AUTO_INCREMENT = 100`（或更高）避免與真實員工帳號混淆。保留 id 6~99 給未來擴充（例如拆分 `service_*` 時可以用）。

### 3.3 與多公司架構的關係

搭配 [0307_多公司與帳號設計](0307_多公司與帳號設計.md) 的 `company_ids` 欄位：

| 帳號 | `company_ids` | `default_company_id` | 備註 |
|------|---------------|----------------------|------|
| `system`  | `null` | `null` | 不走公司過濾（以 role 判斷或 middleware 跳過） |
| `service` | `null` | `null` | 同上，介接流程可能跨任一公司 |
| `admin`   | `null` | `null` | 同上，擁有 `super_admin` role |
| `demo`    | `[測試公司 id]` | 測試公司 | 綁定測試資料範圍，避免誤碰正式資料 |
| `reader`  | 依使用者需求 | 依使用者需求 | 搭配只讀角色 |

### 3.4 與 employer_id 的關係

這四個帳號都**不是員工**，一律 `employer_id = null`，不建立 `employees` 子表記錄。與 [0125_外部組織與身份子表架構](0125_外部組織與身份子表架構.md) 「沒有子表 = 不具該身份」的原則一致。

## 4. Seeder 規劃

### 4.1 檔案位置

```
database/seeders/SystemUsersSeeder.php
```

以最早順序執行，確保後續所有 seeder 在建立資料時都能引用 `system.id`。

### 4.2 範例

```php
class SystemUsersSeeder extends Seeder
{
    public function run(): void
    {
        $now = now();

        DB::table('users')->insert([
            [
                'id' => 1,
                'username' => 'system',
                'email' => 'system@local',
                'password' => Hash::make(Str::random(64)),  // 不會被使用，隨機即可
                'is_system' => true,
                'is_loginable' => false,
                'created_at' => $now, 'updated_at' => $now,
            ],
            [
                'id' => 2,
                'username' => 'service',
                'email' => 'service@local',
                'password' => Hash::make(Str::random(64)),  // 不會被使用（未來走 API token）
                'is_system' => true,
                'is_loginable' => false,
                'created_at' => $now, 'updated_at' => $now,
            ],
            [
                'id' => 3,
                'username' => 'admin',
                'email' => env('SEED_ADMIN_EMAIL', 'admin@local'),
                'password' => Hash::make(env('SEED_ADMIN_PASSWORD')),
                'is_system' => true,
                'is_loginable' => true,
                'created_at' => $now, 'updated_at' => $now,
            ],
            [
                'id' => 4,
                'username' => 'demo',
                'email' => 'demo@local',
                'password' => Hash::make('demo1234'),
                'is_system' => true,
                'is_loginable' => true,
                'created_at' => $now, 'updated_at' => $now,
            ],
            [
                'id' => 5,
                'username' => 'reader',
                'email' => 'reader@local',
                'password' => Hash::make('reader1234'),
                'is_system' => true,
                'is_loginable' => true,
                'created_at' => $now, 'updated_at' => $now,
            ],
        ]);

        DB::statement('ALTER TABLE users AUTO_INCREMENT = 100');
    }
}
```

### 4.3 密碼策略

| 帳號 | 密碼來源 | 說明 |
|------|----------|------|
| `system`  | 隨機 64 字元 | 不會被使用（`is_loginable = false`） |
| `service` | 隨機 64 字元 | 不會被使用（`is_loginable = false`，未來走 API token 認證） |
| `admin`   | `.env` 變數 `SEED_ADMIN_PASSWORD` | 禁止 hardcode，部署時由運維設定 |
| `demo`    | 固定可公開的弱密碼 | 展示方便，功能上有限制 |
| `reader`  | 固定可公開的弱密碼 | 同 demo |

### 4.4 Role 對應

搭配 [0104_權限機制](0104_權限機制.md) 的 role 設計：

| 帳號 | role |
|------|------|
| `system`  | `system`（特殊，不走一般權限檢查） |
| `service` | `service`（特殊，僅允許 API 呼叫，不走後台路由） |
| `admin`   | `super_admin` |
| `demo`    | `demo`（限定可操作範圍） |
| `reader`  | `reader`（只讀） |

## 5. 防呆機制

### 5.1 Model Observer

```php
// app/Observers/UserObserver.php

public function deleting(User $user): void
{
    if ($user->is_system) {
        throw new \DomainException("系統內建帳號不可刪除：{$user->username}");
    }
}

public function updating(User $user): void
{
    if (!$user->is_system) return;

    // 系統帳號的關鍵欄位不可改
    $locked = ['username', 'is_system', 'is_loginable'];
    foreach ($locked as $field) {
        if ($user->isDirty($field)) {
            throw new \DomainException("系統內建帳號的 {$field} 不可變更");
        }
    }
}
```

### 5.2 登入限制

```php
// LoginController 或 Auth middleware
if (!$user->is_loginable) {
    throw new \Illuminate\Auth\AuthenticationException('此帳號不允許登入');
}
```

### 5.3 UI 層隱藏

使用者管理介面（Ocadmin）應：
- `is_system = true` 的帳號顯示鎖頭圖示
- 隱藏「刪除」按鈕
- 編輯頁的 `username`、`is_system`、`is_loginable` 欄位設為 readonly

## 6. `getActorId()` Helper

這是最關鍵的設計——**程式碼裡永遠不要 hardcode `system.id` 或 `service.id`**，統一走 helper：

```php
// app/Support/Auth/ActorResolver.php

class ActorResolver
{
    public const ID_SYSTEM  = 1;
    public const ID_SERVICE = 2;

    /**
     * 取得當前操作者的 user id。
     *
     * 優先序：
     * 1. web 登入的真人 → auth()->id()
     * 2. API token 對應的 user（真人或 service 帳號）
     * 3. 明確的業務流程（webhook / 排程 / 匯入） → service (id=2)
     * 4. 不知道是誰（framework / seeder / fallback） → system (id=1)
     */
    public static function id(): int
    {
        if ($id = auth()->id()) {
            return $id;
        }

        return self::ID_SYSTEM;
    }

    /**
     * 明確指定走 service 身份。
     * 用於 webhook handler、排程同步 job、批次匯入指令等。
     */
    public static function asService(callable $callback): mixed
    {
        return app()->instance('actor.override', self::ID_SERVICE)
            ? $callback()
            : $callback();
    }
}

// 全域 helper
function actor_id(): int
{
    return app()->bound('actor.override')
        ? app('actor.override')
        : \App\Support\Auth\ActorResolver::id();
}
```

使用：

```php
// 所有 Model 的 created_by / updated_by 都走這個
$model->created_by = actor_id();

// 明確知道是自動化流程時，標註為 service
ActorResolver::asService(function () use ($payload) {
    Order::create([...]);  // 內部的 actor_id() 會回傳 2 (service)
});
```

**好處**：未來要拆分 `service_web` / `service_hrm_sync` 等細分帳號時，只改 `ActorResolver` 的判斷邏輯，不動 Model、不動資料表。

## 7. 未來演進：`service` 的細分

第一版只有一筆泛用 `service`。當系統開始出現多種「機器對機器」介接流程時，可進一步細分——**但前提是細分有審計價值**，否則維持單一 `service` 即可。

### 7.1 觸發條件

- 官網下單呼叫 POS API
- 打卡機廠商定時匯入資料
- 發票廠商 webhook 回寫
- 多子系統間的資料同步（accounts ↔ hrm ↔ pos）

### 7.2 拆分範例

| 情境 | 身份 |
|------|------|
| Framework / Seeder / 不知道是誰 | `system`（保留） |
| 通用自動化流程（尚未細分） | `service`（保留） |
| 官網 API 建單 | `service_web` |
| 打卡機匯入 | `service_hrm_sync` |
| 發票 webhook | `service_invoice` |
| 會員同步 | `service_sync_accounts` |

### 7.3 是否遷出 `users` 表

長期來看，service 帳號與真人帳號的關注點差異頗大，可考慮獨立：

```
users                // 真人帳號
service_accounts     // 系統對系統的身份（持有 API token）
api_clients          // OAuth client / personal access token 的擁有者
```

理由：
- `users` 的欄位（姓名、員工關聯、公司歸屬）對 service 帳號多半無意義
- 權限模型不同（service 只能走特定 API，不能登入後台）
- 審計需求不同（service 的每一次呼叫都要記 request log，真人不用）

**不過第一版不急著拆**——保留在 `users` 可以讓 `created_by` / `updated_by` 的 FK 統一指向同一張表，查詢簡單；等到 service 帳號超過 5 個、或出現明顯的 schema 衝突（例如需要 `api_scope` 欄位）時再拆。

### 7.4 `system` vs `service` 速查

| | `system` | `service` |
|---|----------|-----------|
| 層級 | 平台底層 fallback | 應用層業務流程 |
| 觸發 | 不知道是誰做的 | 某個明確業務流程 |
| 典型來源 | Seeder / Migration / Queue 無 context | Webhook / 排程 / API 介接 / 批次匯入 |
| 登入 | 不可 | 不可 |
| 認證方式 | 不認證（只作掛名） | 未來走 API token / Sanctum |
| 第一版數量 | 1 個 | 1 個（泛用） |
| 未來數量 | 1 個（不拆） | N 個（按流程拆 `service_*`） |
| 當前資料表 | `users` | `users` |
| 長期資料表 | `users` | 可能遷至 `service_accounts` |

## 8. 實作順序

| 階段 | 內容 |
|------|------|
| 1 | Migration：新增 `is_system` / `is_loginable` 欄位 |
| 2 | `SystemUsersSeeder`：建立四個固定 ID 的全域帳號 |
| 3 | `UserObserver`：防刪除、防改名 |
| 4 | `ActorResolver` / `actor_id()` helper |
| 5 | LoginController：擋 `is_loginable = false` 的帳號 |
| 6 | Ocadmin UI：系統帳號加鎖頭、隱藏刪除鈕 |
| 7 | 所有 Model 的 `created_by` / `updated_by` 改走 `actor_id()` |
