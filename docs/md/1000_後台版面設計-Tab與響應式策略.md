# 1000 後台版面設計：Tab 與響應式策略

> 狀態：議題參考（尚未實作）
>
> 本文件整理後台版面在桌機與手機上的 Tab 選型策略、響應式拆分原則，
> 作為未來 Ocadmin 版面設計的決策依據。

---

## 1. 核心原則

**手機 = 看（View / Inspect），桌機 = 改（Edit / Maintain）**

- 主管用手機主要是「快速查看狀態」，不是做完整編輯
- 完整編輯任務留在桌機，避免誤觸與操作風險
- 這不是妥協，是角色分工——成熟後台系統的常態

---

## 2. 桌機：水平 Tab vs 垂直 Tab

### 2.1 結論

桌機後台編輯場景，**水平 Tab 是預設首選**，垂直 Tab 僅在少數情況才值得採用。

### 2.2 水平 Tab 的優勢

| 優勢 | 說明 |
|------|------|
| 肌肉記憶 | 後台、IDE、瀏覽器都是「上面切頁籤、下面做事」，零學習成本 |
| 不浪費橫向空間 | 桌機表單本身就需要寬度（label + input + 說明 + 多欄並排），垂直 Tab 會永久吃掉左側 |
| 適合同一物件多面向 | 商品的「基本 / 多語 / 選項」是「視角切換」，正是水平 Tab 存在的目的 |
| 工程面好處理 | 錯誤提示集中、驗證失敗後跳回 Tab、表單一次送出 |

### 2.3 何時改用垂直 Tab

| 條件 | 說明 | 範例 |
|------|------|------|
| Tab 很多（6~10+） | 水平 Tab 會擠、換行、變醜，垂直清單反而穩 | 基本 / 多語 / 價格 / 庫存 / SEO / 圖片 / 權限 / 擴充 |
| Tab 是模組級別 | 比較像側邊選單，而非 Tab | 帳戶設定 / 權限管理 / 系統參數 |
| 內容高度差異極大 | 垂直 Tab 切換時視覺較穩，畫面不會跳動 | A tab 3 欄位 vs B tab 200 行表格 |

### 2.4 速記判斷法

> **Tab 是拿來「平行切換」的，不是拿來「當導覽列」的。**
>
> - 平行視角 → 水平 Tab
> - 導覽模組 → 垂直 Tab

### 2.5 商品編輯實例判斷

| 條件 | 情況 | 結論 |
|------|------|------|
| Tab 數量 | 少（3~4 個） | 水平 |
| 編輯密度 | 高 | 水平 |
| 使用頻率 | 長時間操作 | 水平 |
| 操作者 | 後台人員 | 水平 |

商品編輯（基本資料 / 多語 / 選項）→ **100% 選水平 Tab**。

## 3. 手機：查看導向的版面設計

### 3.1 為什麼手機不適合用 Tab

不論水平或垂直 Tab，在手機上都有根本性問題：

| Tab 類型 | 問題 |
|---------|------|
| 水平 Tab | 文字被截斷、太多只能橫向捲動、手指誤觸 |
| 垂直 Tab | 鍵盤一出來就被擠爆、仍要記得「我在哪個 Tab」、驗證錯誤跳 Tab 很惱人 |

**水平 Tab 在手機上的存活條件**（非常嚴苛）：

- Tab 數量 ≤ 3 個
- 每個標籤 ≤ 2 個中文字
- 不需要橫向捲動
- 有明顯 active 狀態

超過此條件就應該換設計。

### 3.2 手機查看頁的黃金結構：Card + Section

**「資訊卡（Card）+ 章節化（Section）」是手機後台查看的最穩解法。**

結構分三層：

#### 頂部：關鍵摘要（Summary Card）

螢幕一打開就看得到，5 秒內掌握重點。

```
商品名稱
狀態：上架中
售價：120
```

#### 中段：資訊卡（Cards）

每一張卡就是桌機上一個 Tab 的內容，改成只讀呈現。

```
[ 基本資料 ]          [ 多語 ]              [ 選項 ]
SKU：A123            中文：招牌潤餅便當      加料：培根、炸蝦
分類：便當            英文：Signature Wrap   飲料：紅茶、奶茶
```

一張卡 = 一個主題，滑動即可，不用思考導覽。

#### 底部（可選）：低風險操作

僅限簡單切換型操作，且須固定底部、明確顏色、有確認提示：

- 上架 / 下架切換
- 啟用 / 停用

**不給手機做的事：** 全表單編輯、跨 Tab 修改、高風險設定。

### 3.3 資訊過多時的處理

| 解法 | 做法 | 適用場景 |
|------|------|---------|
| 卡片限行 | 卡片內最多 5 行，多的顯示「…顯示更多」 | 一般情況 |
| 進子頁（只讀） | 點進去看完整列表，仍然只讀 | 選項多、資料量大 |

### 3.4 桌機 Tab 對照手機的轉換表

| 原本（桌機 Tab） | 手機最佳呈現 |
|----------------|-----------|
| 基本資料 Tab | 基本資料卡 |
| 多語 Tab | 多語卡 |
| 選項 Tab | 選項卡 |
| 切 Tab | 滑動 |
| 表單 | 只讀欄位 |

## 4. 響應式策略：成熟版 RWD

### 4.1 不是「完全不共用 RWD」，是「不共用同一個操作設計」

| 做法 | 說明 |
|------|------|
| 錯誤做法 | 桌機一套 Blade，手機再寫一套完全不同的頁面，路由、邏輯、驗證全分叉 → 維護地獄 |
| 正確做法 | 同一個功能、同一個 route、同一份資料，但用「不同 Layout + 不同 UI 表現」 |

> **共用「資料結構與商業邏輯」，分開「呈現方式與可做的事」。**

### 4.2 天真版 vs 成熟版 RWD

| | 天真版 RWD | 成熟版 RWD |
|--|----------|----------|
| 做法 | 同一個畫面，用 `hidden md:block` 把 Tab 硬藏起來 | 同一功能，不同 Layout，不同互動假設 |
| 本質 | 解決排版問題 | 解決使用者在不同裝置上「目的不同」的問題 |

### 4.3 實際落地範例

同一個 route：`GET /products/{id}`

**桌機（Desktop Layout）：**
- 水平 Tab、可編輯、表單導向

**手機（Mobile Layout）：**
- 卡片式、只讀、看重點

### 4.4 Laravel 技術實作思路

**Blade 方式：**

```
layouts/
├── desktop.blade.php
└── mobile.blade.php
```

Controller 判斷裝置後回傳不同 view：

```php
return view(
    isMobile() ? 'products.show_mobile' : 'products.show_desktop',
    $data
);
```

或共用 content partial，只換外殼 Layout。

**Inertia / 前後端分離方式：**

同一個 page，判斷 viewport 後 render 不同 component：

```jsx
isMobile ? <MobileView /> : <DesktopView />
```

### 4.5 工程面的分離好處

```
GET  /products/{id}       → 查看（手機友善）
GET  /products/{id}/edit  → 編輯（桌機）
PUT  /products/{id}/basic        → 基本資料
PUT  /products/{id}/translations → 多語
PUT  /products/{id}/options      → 選項
```

- 權限可拆：`product.view` / `product.edit`
- Validation 不打架
- 出錯不用跳 Tab
- API 可重用（App / BI / 報表都能吃 view 那一套）

## 5. 決策準則速查

### 5.1 手機版是否需要該功能？

> 如果你會說「這個頁面在手機上很累」，那它就不該存在於手機。

### 5.2 是否該拆 Layout？

> 只要手機版「刻意拿掉某些功能」，那它就不該只是桌機的 RWD 縮小版。

### 5.3 Tab 內容是否值得有獨立 URL？

> 如果一個 Tab 裡的內容「本來就值得有 URL」，那它在手機上就該是一個獨立頁面。

### 5.4 Tab 的本質判斷

> Tab 是為了「快速切換視角」，不是為了「幫表單分段」。
>
> 商品的「基本 / 多語 / 選項」本質是**分段表單**，不是分頁瀏覽。

## 6. UI 元件選擇參考

| 元件 | 建議 |
|------|------|
| Card | Tailwind / Bootstrap 皆可 |
| Section title | 清楚、不要花俏 |
| 字級 | 標題稍大、資料正常 |
| 行距 | 手機上比桌機寬一點 |

> **手機後台不是操作介面，是「資訊儀表板」。**
