## 一、概述

後台側欄選單需依使用者權限動態顯示：使用者只能看到自己有權限的選單項目。本文件記錄兩種選單方案的完整設計，兩套並存於 `MenuComposer` 中，透過設定切換。

- **方案 A — Code Driven**：選單以 PHP hardcoded array 定義，MenuComposer 收集後依權限過濾。
- **方案 B — DB Driven**：選單存入獨立的 `sys_menus` / `sys_menu_translations` 表，與權限表分離，透過 `permission_name` 欄位關聯權限做可見性判斷。

**切換方式：** `config/vars.php` 的 `menu_driver` 設定，或 `.env` 設定 `MENU_DRIVER`：

| 值 | 說明 |
|---|---|
| `database` | 從 `sys_menus` 表讀取（預設） |
| `code` | 從 MenuComposer hardcoded array 讀取 |

本範例專案預設使用 DB 驅動；將本專案做為基底建立具體專案時，可依需求選擇任一方案。兩種方案的優劣比較見 [§4 方案評估比較](#四方案評估比較)。

### 1.1 選單權限慣例：`*.menu` action

選單可見性用獨立的 `*.menu` 權限後綴，與資料存取（`.access` / `.access_all`）解耦：

| Action | 用途 | 由誰判定 |
|---|---|---|
| `{resource}.menu` | 選單是否顯示在 sidebar | MenuComposer |
| `{resource}.access` | 列表 + 單筆檢視（基礎資料權限） | Controller `authorize()` |
| `{resource}.access_all` | 看全部（不限 created_by） | Controller 內部 scope |
| `{resource}.modify` / `.delete` / ... | 各操作 | Controller `authorize()` |

**為何分開？** 三者語意不同，不該強制蘊含：

- 純 UI 可見性（`.menu`）vs 資料權限（`.access`）拆開 → 角色設計時各自勾選，意圖明確
- 不需在程式碼裡寫「access_all 蘊含 access」之類的隱式邏輯
- 權限管理 UI 上一個 checkbox 一件事

**規則：**

1. 選單檔／選單表的 `permission` 一律填 `*.menu`
2. Controller `authorize()` 一律用 `*.access` / `*.modify` / `*.delete` / ...，**不**用 `*.menu`
3. Role 設定 UI 可加前端規則：勾 `*.access_all` 自動連帶勾 `*.access`（後者為基礎必選）
4. 父層分組節點（無 `href`、無 `route`）不需 `permission`；子項全被過濾時自動隱藏

## 二、方案 A — Code Driven（MenuComposer）

選單以 PHP 檔案定義，每個模組一個 Menu class，MenuComposer 收集後依使用者權限過濾。

```
app/Portals/Ocadmin/
├── Core/ViewComposers/MenuComposer.php    ← 收集 + 權限過濾
├── Modules/
│   ├── Catalog/CatalogMenu.php            ← 商品型錄
│   ├── Inventory/InventoryMenu.php        ← 庫存管理
│   ├── Master/MasterMenu.php              ← 主檔管理
│   ├── Finance/FinanceMenu.php            ← 財務會計
│   └── Hrm/HrmMenu.php                   ← 人事管理
```

### 2.1 Menu Class 規範

每個模組在自己的目錄下定義 Menu class，實作 `items()` 靜態方法，回傳單一選單群組：

```php
// app/Portals/Ocadmin/Modules/Finance/FinanceMenu.php

namespace App\Portals\Ocadmin\Modules\Finance;

class FinanceMenu
{
    public static function items(): array
    {
        return [
            'id'         => 'menu-finance',
            'icon'       => 'fa-solid fa-coins',
            'name'       => '財務會計',
            // 父層分組節點通常無 permission，由子項向上推導可見性
            'href'       => '',
            'children'   => [
                [
                    'name'       => '付款',
                    'icon'       => '',
                    'permission' => 'admin.finance.payment.menu',
                    'href'       => route('lang.ocadmin.finance.payment.index'),
                    'children'   => [],
                ],
                [
                    'name'       => '退款',
                    'icon'       => '',
                    'permission' => 'admin.finance.refund.menu',
                    'href'       => route('lang.ocadmin.finance.refund.index'),
                    'children'   => [],
                ],
            ],
        ];
    }
}
```

**欄位說明：**

| 欄位 | 必填 | 說明 |
|------|------|------|
| `id` | 僅 L1 | HTML id，用於 sidebar 展開/收合狀態 |
| `icon` | 僅 L1 | FontAwesome class |
| `name` | 是 | 顯示名稱 |
| `permission` | 否 | 對應的選單權限名稱，慣例填 `{resource}.menu`；無此欄位則不做權限檢查 |
| `href` | 是 | 連結，空字串表示純分組節點 |
| `children` | 是 | 子項目陣列 |

### 2.2 MenuComposer 收集與過濾

```php
// app/Portals/Ocadmin/Core/ViewComposers/MenuComposer.php

use App\Portals\Ocadmin\Modules\Catalog\CatalogMenu;
use App\Portals\Ocadmin\Modules\Inventory\InventoryMenu;
use App\Portals\Ocadmin\Modules\Master\MasterMenu;

class MenuComposer
{
    /**
     * 選單註冊表 — 順序即為 sidebar 顯示順序
     */
    protected array $menuClasses = [
        // DashboardMenu::class,
        CatalogMenu::class,
        InventoryMenu::class,
        MasterMenu::class,
        // MemberMenu::class,
        // FinanceMenu::class,
        // SystemMenu::class,
    ];

    public function compose(View $view): void
    {
        $view->with('sys_menus', $this->buildMenus());
    }

    protected function buildMenus(): array
    {
        $user = auth()->user();

        return collect($this->menuClasses)
            ->map(fn ($class) => $class::items())
            ->map(fn ($menu) => $this->filterByPermission($menu, $user))
            ->filter()         // 移除 null（整個群組無權限）
            ->values()
            ->toArray();
    }

    /**
     * 遞迴過濾：移除無權限的項目，子項全空的父層也移除
     */
    protected function filterByPermission(array $item, $user): ?array
    {
        // 檢查項目本身的權限
        if (!empty($item['permission']) && !$user->can($item['permission'])) {
            return null;
        }

        // 遞迴過濾子項目
        if (!empty($item['children'])) {
            $item['children'] = collect($item['children'])
                ->map(fn ($child) => $this->filterByPermission($child, $user))
                ->filter()
                ->values()
                ->toArray();

            // 分組節點（href 為空）且子項全被過濾 → 隱藏父層
            if (empty($item['children']) && empty($item['href'])) {
                return null;
            }
        }

        return $item;
    }
}
```

**過濾邏輯：**

1. 項目有 `permission` 欄位 → 使用者無此權限則移除
2. 項目無 `permission` 欄位 → 不檢查，直接保留
3. 子項目全部被移除且父層 `href` 為空 → 父層也移除（避免空的分組節點）
4. `developer` 的 `can()` 永遠回傳 `true`（見 [10007 §11.5](10007_權限機制.md#十一使用者權限快取)），選單全部可見；`super_admin` 依角色指派的權限決定

**父層可見性規則：** 父層選單（如「系統管理」、「財務會計」）通常不設 `permission`，其可見性由子項向上推導決定：

- 至少一個子項有權限 → 父層**顯示**
- 所有子項都無權限 → 子項全被過濾 → 父層自動**隱藏**

因此不需要為父層選單額外定義整段群組的 `*.menu`，子項各自的 `{resource}.menu` 已足夠控制整個選單群組的可見性。

### 2.3 新增模組 Checklist

- [ ] 在模組目錄下建立 `{Module}Menu.php`，實作 `items()` 方法
- [ ] 每個子項目填入對應的 `permission`（慣例填 `{resource}.menu`）
- [ ] 在 `MenuComposer::$menuClasses` 中註冊，插入期望的排序位置
- [ ] 確認對應的 `*.menu` 權限已在 `AclPermissionSeeder` 中定義

## 三、方案 B — DB Driven（獨立選單表）

選單存入獨立的 `sys_menus` / `sys_menu_translations` 表，不與 `acl_permissions` 混用。透過 `permission_name` 欄位關聯權限做可見性判斷。

### 3.1 設計原則

選單與權限各司其職：

| | `sys_menus` 表 | `acl_permissions` 表 |
|---|---|---|
| **職責** | 定義 sidebar 的結構、順序、連結 | 定義功能權限（access / modify / delete） |
| **欄位** | portal、icon、route_name、href、sort_order | name、guard_name |
| **關聯方式** | `menus.permission_name` → `acl_permissions.name` | — |

**不混用的好處：**

- `acl_permissions` 不會多出 `icon`、`href`、`sort_order` 等選單專用欄位
- Spatie 的 `hasPermissionTo()` / `can()` 不會撈到選單節點
- 選單可以有不需權限檢查的純分組節點，不必塞進權限表

### 3.2 資料表結構

```
menus                                  menu_translations
├── id                                 ├── id
├── portal ──→ 'ocadmin','www','pos'   ├── menu_id (FK → menus)
├── parent_id (FK → menus, nullable)   ├── locale
├── permission_name (nullable) ──→     ├── display_name
│   acl_permissions.name               └── unique(menu_id, locale)
├── route_name (nullable)
├── href (nullable)
├── icon (nullable)
├── sort_order (default: 0)
├── is_active (default: true)
└── timestamps
```

**欄位說明：**

| 欄位 | 說明 |
|------|------|
| `portal` | 所屬應用入口：`ocadmin`（後台）、`www`（官網）、`pos`（POS）等，對應 `config/portals.php` |
| `parent_id` | 自我關聯，`null` 表示 L1 根節點 |
| `permission_name` | 關聯到 `acl_permissions.name`，慣例填 `{resource}.menu`；`null` 表示不檢查權限（純分組節點，可見性由子項推導） |
| `route_name` | 本系統的 named route，MenuComposer 用 `route()` 動態產生 URL（含語系前綴） |
| `href` | 外部連結或靜態 URL，直接輸出到前端 |
| `icon` | FontAwesome class，僅 L1 根節點使用 |
| `sort_order` | 同層排序，數字越小越前面 |
| `is_active` | 停用選單項目而不刪除 |

**`route_name` 與 `href` 的優先順序：**

| route_name | href | 結果 |
|---|---|---|
| 有值 | — | `route($route_name)` 產生 URL（本系統路由） |
| null | 有值 | 直接輸出 `href`（外部連結） |
| null | null | 純分組節點（可展開收合的父層） |

### 3.3 樹狀結構範例

以 Ocadmin 後台為例：

├── catalog.product                  (L2, permission=admin.catalog.product.menu, route=lang.ocadmin.catalog.product.index)

### 3.5 Menu Model

```php
// app/Models/Menu.php

namespace App\Models;

use App\Traits\HasTranslation;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Menu extends Model
{
    use HasTranslation;

    protected $fillable = [
        'portal', 'parent_id', 'permission_name',
        'route_name', 'href', 'icon', 'sort_order', 'is_active',
    ];

    protected string $translationModel = MenuTranslation::class;
    protected array $translatedAttributes = ['display_name'];

    protected $with = ['translation'];

    public function parent(): BelongsTo
    {
        return $this->belongsTo(Menu::class, 'parent_id');
    }

    public function children(): HasMany
    {
        return $this->hasMany(Menu::class, 'parent_id')
            ->where('is_active', true)
            ->orderBy('sort_order');
    }

    /**
     * 產生 URL：route_name 優先，其次 href，都沒有回傳空字串
     */
    public function resolveUrl(): string
    {
        if ($this->route_name) {
            return route($this->route_name);
        }

        return $this->href ?? '';
    }

    /**
     * 轉為前端 sidebar 所需的 array 格式
     */
    public function toMenuItem(): array
    {
        return [
            'id'         => 'menu-' . $this->id,
            'icon'       => $this->icon ?? '',
            'name'       => $this->display_name,
            'permission' => $this->permission_name,
            'href'       => $this->resolveUrl(),
            'children'   => $this->children->map(fn ($child) => $child->toMenuItem())->toArray(),
        ];
    }
}
```

### 3.6 MenuComposer（DB 版）

```php
// app/Portals/Ocadmin/Core/ViewComposers/MenuComposer.php

use App\Models\Menu;

class MenuComposer
{
    public function compose(View $view): void
    {
        $view->with('sys_menus', $this->buildMenus());
    }

    protected function buildMenus(): array
    {
        $user = auth()->user();

        // 撈該 portal 的 L1 根節點（含遞迴子項）
        $menus = Menu::with('children.children')  // eager load 2~3 層
            ->where('portal', 'ocadmin')
            ->whereNull('parent_id')
            ->where('is_active', true)
            ->orderBy('sort_order')
            ->get();

        return $menus
            ->map(fn ($menu) => $menu->toMenuItem())
            ->map(fn ($item) => $this->filterByPermission($item, $user))
            ->filter()
            ->values()
            ->toArray();
    }

    /**
     * 遞迴過濾：移除無權限的項目，子項全空的父層也移除
     */
    protected function filterByPermission(array $item, $user): ?array
    {
        if (!empty($item['permission']) && !$user->can($item['permission'])) {
            return null;
        }

        if (!empty($item['children'])) {
            $item['children'] = collect($item['children'])
                ->map(fn ($child) => $this->filterByPermission($child, $user))
                ->filter()
                ->values()
                ->toArray();

            if (empty($item['children']) && empty($item['href'])) {
                return null;
            }
        }

        return $item;
    }
}
```

**父層可見性規則：** 父層選單（如「商品型錄」）的 `permission_name` 設為 `null`，不做權限檢查，可見性由子項向上推導：

```
商品型錄  (permission_name = null, 純分組節點)
├── 商品      (permission_name = admin.catalog.product.menu)
├── 選項      (permission_name = admin.catalog.option.menu)
└── 規格      (permission_name = admin.catalog.specification.menu)
```

- 使用者有任一子項 `*.menu` 權限 → 「商品型錄」**顯示**
- 使用者三個都沒有 → 子項全被過濾 → 「商品型錄」自動**隱藏**

因此不需為父層選單額外定義群組權限（如 `admin.catalog.*.menu` 整段），子項各自的 `*.menu` 已足夠控制整個選單群組的可見性。`developer` 的 `can()` 永遠回傳 `true`（見 [10007 §10.5](10007_權限機制.md#十使用者權限快取)），選單全部可見；`super_admin` 依角色指派的權限決定。

### 3.7 Seeder 範例

```php
// database/seeders/MenuSeeder.php

use App\Models\Menu;

class MenuSeeder extends Seeder
{
    public function run(): void
    {
        $portal = 'ocadmin';

        // L1: 商品型錄
        $catalog = Menu::updateOrCreate(

        // ... 其餘模組依此類推
    }

```

### 3.9 多 Portal 支援

同一張 `sys_menus` 表透過 `portal` 欄位區分不同應用的選單：

| portal | 應用 | 說明 |
|--------|------|------|
| `ocadmin` | 後台管理 | 完整的 sidebar 選單樹 |
| `www` | 官網 ApiWww | 前台導覽列 |
| `pos` | POS 系統 ApiPos | POS 功能選單 |

每個 Portal 的 MenuComposer 只撈自己的 `portal`：

```php
// Ocadmin
Menu::where('portal', 'ocadmin')->whereNull('parent_id')...

// POS
Menu::where('portal', 'pos')->whereNull('parent_id')...
```

不同 Portal 的選單完全獨立，不會互相干擾。同一個權限（如 `admin.catalog.product.access`）可被多個 Portal 的選單節點引用。

## 四、方案 C — File Driven（resources/menus）

> 2026-04-19 新增。branchSunline、branchAdminlte、branchHuabing 採用此方案。

選單以 PHP 檔案定義，**集中放置於 `resources/menus/`**，每個選單區塊一個檔案。與方案 A 的差異在於：選單與模組解耦，不綁定模組資料夾結構。

### 4.1 設計理念

- **模組是程式碼的組織方式，選單是使用者的導航方式**，兩者不該強制 1:1
- 同一個作業可以出現在不同選單（如「供應商」同時出現在庫存和主檔）
- 新增選單不影響模組資料夾結構

### 4.2 檔案結構

```
app/Portals/Ocadmin/
├── Core/ViewComposers/MenuComposer.php    ← 掃描 menus/*.php + 權限過濾
├── resources/
│   ├── views/
│   └── menus/                             ← 選單定義（與 views 同級）
│       ├── dashboard.php     (sort: 100)
│       ├── dealer.php        (sort: 200)
│       ├── catalog.php       (sort: 300)
│       ├── sale.php          (sort: 400)
│       └── system.php        (sort: 900)
```

### 4.3 menu.php 格式

每個檔案 return 一個選單群組 array：

```php
// resources/menus/catalog.php
<?php

return [
    'id'       => 'menu-catalog',                    // HTML id（僅 L1）
    'icon'     => 'fa-solid fa-book',                // icon class（僅 L1）
    'name'     => 'admin/common/admin_menu.catalog',  // 翻譯 key，自動 __()
    'sort'     => 300,                                // 選單區塊間的排序
    'children' => [
        [
            'name'       => 'admin/common/admin_menu.catalog_option',
            'route'      => 'lang.ocadmin.catalog.options.index',  // 自動轉 href
            'permission' => 'admin.catalog.option.menu',           // 慣例 *.menu
        ],
    ],
];
```

**欄位說明：**

| 欄位 | 必填 | 說明 |
|------|------|------|
| `id` | 僅 L1 | HTML id，用於 sidebar 展開/收合 |
| `icon` | 僅 L1 | Icon class |
| `name` | 是 | 翻譯 key（MenuComposer 自動 `__()`） |
| `route` | 否 | Named route，MenuComposer 自動 `route()` 轉為 href |
| `href` | 否 | 直接 URL（route 優先於 href） |
| `sort` | 僅 L1 | 選單區塊排序，數字越小越前 |
| `permission` | 否 | 選單權限名稱，慣例填 `{resource}.menu`；無此欄位則不檢查 |
| `children` | 否 | 子項目陣列 |

### 4.4 選單翻譯

選單顯示名稱透過 Laravel 翻譯檔管理，與選單結構分離：

```php
// lang/zh_Hant/admin/common/admin_menu.php
return [
    'dashboard' => '儀表板',
    'catalog' => '商品型錄',
    'catalog_option' => '選項',
    // ...
];
```

翻譯 key 以 `{選單區塊}_{項目}` 命名。若同一作業出現在不同選單，各自定義 key（如 `inventory_supplier`、`master_supplier`），值可以相同。

### 4.5 條件式選單

menu.php 本身就是 PHP 檔案，可直接寫邏輯：

```php
// resources/menus/system.php
<?php

use App\Services\System\Database\SchemaParserService;

$children = [
    // ... 固定選單項目
];

// 條件式：需通過驗證才顯示
if (SchemaParserService::validateParser()) {
    $children[] = [
        'name'     => 'admin/common/admin_menu.system_taxonomy',
        'children' => [ /* ... */ ],
    ];
}

return [
    'id'       => 'menu-system',
    'icon'     => 'fa-solid fa-cog',
    'name'     => 'admin/common/admin_menu.system',
    'sort'     => 900,
    'children' => $children,
];
```

### 4.6 MenuComposer

```php
class MenuComposer
{
    protected function buildMenus(): array
    {
        $user = auth()->user();
        $menusPath = app_path('Portals/Ocadmin/resources/menus');

        // 收集所有選單檔案
        $raw = [];
        foreach (glob("{$menusPath}/*.php") as $file) {
            $menu = require $file;
            if (is_array($menu)) {
                $raw[] = $menu;
            }
        }

        // 依 sort 排序
        usort($raw, fn ($a, $b) => ($a['sort'] ?? 500) <=> ($b['sort'] ?? 500));

        // 解析並過濾
        return collect($raw)
            ->map(fn ($item) => $this->resolveItem($item))
            ->map(fn ($item) => $this->filterByPermission($item, $user))
            ->filter()
            ->values()
            ->toArray();
    }

    protected function resolveItem(array $item): array
    {
        $item['name'] = __($item['name']);  // 翻譯
        $item['icon'] = $item['icon'] ?? '';
        $item['id'] = $item['id'] ?? '';

        if (!empty($item['route'])) {
            try {
                $item['href'] = route($item['route']);
            } catch (\Exception $e) {
                $item['href'] = '#';  // route 不存在時 fallback
            }
        } else {
            $item['href'] = $item['href'] ?? '';
        }

        if (!empty($item['children'])) {
            $item['children'] = array_map(
                fn ($child) => $this->resolveItem($child), $item['children']
            );
        } else {
            $item['children'] = [];
        }

        return $item;
    }

    protected function filterByPermission(array $item, $user): ?array
    {
        // 走 Gate::forUser()->check() 確保 Gate::before 生效
        if (!empty($item['permission'])
            && !Gate::forUser($user)->check($item['permission'])) {
            return null;
        }
        // ... 遞迴過濾子項（同方案 A/B）
    }
}
```

**注意：** 權限檢查使用 `Gate::forUser($user)->check()` 而非 `$user->can()`，因為 Spatie Permission 的 `can()` 不經過 `Gate::before`，會導致 `super_admin` 的放行邏輯失效。

### 4.7 新增選單 Checklist

- [ ] 在 `resources/menus/` 下新增或修改 `.php` 檔案
- [ ] 設定 `sort` 值控制選單區塊順序
- [ ] 每個子項填入 `route`（named route）和 `permission`（慣例 `{resource}.menu`）
- [ ] 在 `lang/*/admin/common/admin_menu.php` 新增翻譯 key
- [ ] 確認對應的 `*.menu` 權限已在 `AclPermissionSeeder` 中定義

## 五、方案評估比較

### 5.1 三方案比較

| | A. Module Menu class | B. DB Driven | C. File Driven（resources/menus） |
|---|---|---|---|
| **選單定義位置** | 各模組 Menu class | `sys_menus` 表 | `resources/menus/*.php` |
| **選單與模組關係** | 綁定（一模組一 Menu class） | 解耦 | 解耦 |
| **修改方式** | 改 code + deploy | 改 DB（UI 或 seeder） | 改檔案 + deploy |
| **可讀性** | 散落多個 class | 要查 DB | 集中一個資料夾，一目了然 |
| **DB query** | 0 | 1 次（可快取） | 0 |
| **後台 UI 管理** | 不支援 | 支援 | 不支援 |
| **新增選單** | 寫 class + 註冊到 MenuComposer | 寫 seeder + 跑 seed | 新增 .php 檔（自動掃描） |
| **多語支援** | 翻譯檔 | 翻譯表 | 翻譯檔 |
| **條件式選單** | class 內寫邏輯 | DB 欄位控制 | 檔案內寫 PHP 邏輯 |
| **同一作業多選單** | 需跨模組引用 | 多筆記錄引用同 route | 多個檔案引用同 route + 各自翻譯 key |
| **適合場景** | 選單與模組嚴格 1:1 | 大型系統、需後台管理 | 中小型系統、選單結構靈活 |

### 5.2 效能比較

三種方案的效能瓶頸都在**權限過濾**而非選單來源。PHP 檔案的 OPcache、DB 的 query cache 成本都很低，真正的成本是遍歷每個節點呼叫權限判斷。

| 做法 | 每次請求的成本 |
|---|---|
| **A/C（檔案）** | 載入檔案（OPcache）+ 遍歷所有節點做權限判斷 |
| **B（DB）+ 角色組合快取** | 第一次：1 次 DB query + 遍歷過濾 → 存快取。之後：1 次 cache hit |

系統規模小時三者無感差異；規模大時 B + 快取的優勢明顯。A 和 C 也可以加快取，但等於自己重新實作角色組合快取機制。

### 5.3 結論

**branchMain 採用方案 B（DB 驅動）**，搭配 code fallback（`MENU_DRIVER` 切換）。適合需要後台 UI 管理選單、多 Portal 支援的場景。

**具體專案（branchSunline、branchAdminlte、branchHuabing）採用方案 C（File Driven）**。選單與模組解耦，集中管理於 `resources/menus/`，結構清晰且無需維護 DB seeder。

> **備註：** 不論哪種方案，選單維護都是開發人員的工作——選單必須有對應的程式（Controller、Route）才能運作。

> **注意（Gate::before）：** 權限檢查應使用 `Gate::forUser($user)->check()` 而非 `$user->can()`。Spatie Permission 的 `can()` 不經過 Laravel `Gate::before`，會導致 `super_admin` / `developer` 的放行邏輯失效。
