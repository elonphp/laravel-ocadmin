## 1. 概述與啟用開關

2FA 為可選安全模組，透過 `.env` 全域開關控制：

```env
# 預設關閉 — 系統初期或小規模部署不需要 2FA
TWO_FACTOR_ENABLED=false
```

### 前提條件

> **2FA 依賴 UserDevice 模組。** 2FA 的觸發時機（新裝置、裝置信任到期、閒置過久）皆需依據裝置記錄判斷。
> 啟用 2FA 時，UserDevice 模組必須同時安裝。

合法組合：

| 組合 | 是否可行 | 說明 |
|------|---------|------|
| UserDevice 單獨 | 可以 | 裝置列表、遠端登出獨立有價值 |
| 2FA + UserDevice | 可以 | 完整的兩步驟驗證 |
| 2FA 單獨 | **不行** | 無裝置記錄，無法判斷觸發時機 |

### 開關行為

| 狀態 | 行為 |
|------|------|
| `TWO_FACTOR_ENABLED=false`（預設） | 完全不觸發 2FA 流程，登入帳密正確即完成 |
| `TWO_FACTOR_ENABLED=true` | 所有使用者必須完成 2FA 設定，未設定者登入後強制導向設定頁 |

### 設定方式

```php
// config/vars.php（或 config/auth.php）
'two_factor' => [
    'enabled' => env('TWO_FACTOR_ENABLED', false),
    'recovery_codes_count' => 8,                          // 恢復碼數量
],
```

## 2. 技術方案選型

### 方案比較

| 方案 | 優點 | 缺點 | 適用性 |
|------|------|------|--------|
| **Laravel Fortify** | 官方套件、內建 TOTP + recovery codes + 密碼確認、持續維護 | 需排除不用的功能（registration 等） | **推薦** |
| **pragmarx/google2fa-laravel** | 輕量、專注 TOTP | 需自行實作 recovery codes 等 | 備選 |
| **自建** | 完全掌控 | 開發成本高、需處理安全細節 | 不建議 |

### 推薦方案：Laravel Fortify

- 官方維護，與 Sanctum 搭配良好
- 內建 TOTP 驗證、recovery codes、密碼確認
- 不需要 Breeze / Jetstream scaffold，只使用 Fortify 後端邏輯
- 本專案自行撰寫前端，Fortify 僅提供 API / 後端流程

### Fortify 啟用功能

```php
// config/fortify.php
'features' => [
    Features::twoFactorAuthentication([
        'confirm' => true,        // 需輸入驗證碼確認綁定
        'confirmPassword' => true, // 啟用 2FA 前需確認密碼
    ]),
],
```

## 3. TOTP 驗證流程

### 登入流程（2FA 整合）

```
使用者提交 email + password
  → Auth::attempt() 帳密驗證
    → 失敗 → 回傳錯誤
    → 成功 → 檢查 TWO_FACTOR_ENABLED（.env 開關）
      → false → 直接登入成功，redirect dashboard
      → true → 使用者已設定 2FA？（two_factor_confirmed_at 不為 null）
        → 否 → 暫存登入狀態，強制導向 2FA 設定頁（/admin/two-factor/setup）
        → 是 → 此裝置為可信任裝置？（user_devices.trusted_until > now）
          → 是 → 直接登入成功
          → 否（新裝置 / 信任到期 / 閒置過久）→ 暫存登入狀態，顯示 TOTP 輸入頁
            → 使用者輸入 6 位數驗證碼
              → 正確 → 登入成功（可選：信任此裝置）
              → 錯誤 → 提示重試（連續錯誤可加入限速）
            → 使用者輸入恢復碼
              → 正確 → 該碼作廢，登入成功
              → 錯誤 → 提示重試
```

### 2FA 觸發時機

2FA 透過 UserDevice 模組的裝置記錄判斷是否需要觸發，以下任一條件成立即觸發：

| 觸發條件 | 判斷依據 | 說明 |
|----------|---------|------|
| **新裝置** | `user_devices` 中無匹配的 `device_fingerprint` | 首次在此裝置登入 |
| **信任到期** | `user_devices.trusted_until` ≤ now 或為 null | 信任期限已過 |
| **閒置過久** | `user_devices.last_active_at` 超過設定天數 | 裝置長期未使用，需重新驗證 |

> 可信任裝置機制詳見 [10009_UserDevice 裝置管理](10009_UserDevice裝置管理.md#5-可信任裝置2fa-整合)。

### 暫存登入狀態

帳密驗證成功但 2FA 尚未通過時，不建立完整 session，而是暫存使用者 ID 到 session：

```php
// 帳密通過，暫存 user ID
session()->put('two_factor:user_id', $user->id);

// 2FA 通過後，正式登入
Auth::loginUsingId(session()->pull('two_factor:user_id'));
session()->regenerate();
```

此設計確保帳密通過但 2FA 未通過時，使用者無法存取受保護頁面。

## 4. 2FA 初始設定流程

### 設定步驟

```
1. 使用者進入 2FA 設定頁（首次登入強制 / 設定頁主動啟用）
  → 確認密碼（security confirmation）

2. 系統產生 TOTP secret
  → 儲存加密後的 secret 至 users.two_factor_secret
  → 產生 QR Code（含 otpauth:// URI）

3. 使用者用 Authenticator App 掃描 QR Code
  → Google Authenticator / Authy / Microsoft Authenticator 等

4. 使用者輸入 App 顯示的 6 位數驗證碼
  → 驗證成功 → 寫入 users.two_factor_confirmed_at
  → 驗證失敗 → 提示重新掃描或重試

5. 產生 recovery codes（8 組）
  → 一次性顯示，要求使用者安全保存
  → 加密後儲存至 users.two_factor_recovery_codes
  → 此頁離開後無法再次查看（僅能重新產生）
```

### QR Code 內容

```
otpauth://totp/{app_name}:{user_email}?secret={base32_secret}&issuer={app_name}
```

- `app_name`：應用名稱（如 `config('app.name')`）
- `user_email`：使用者信箱
- `base32_secret`：TOTP 密鑰（Base32 編碼）

## 5. 恢復碼（Recovery Codes）

### 設計規則

| 項目 | 規則 |
|------|------|
| 數量 | 8 組 |
| 格式 | `xxxxx-xxxxx`（10 位英數字，中間以 `-` 分隔） |
| 使用方式 | 每組一次性使用，使用後立即作廢 |
| 儲存 | 加密後存入 `users.two_factor_recovery_codes`（JSON 陣列） |
| 顯示 | 僅在產生時顯示一次，使用者須自行保存 |

### 恢復碼生命週期

```
產生 → 使用者保存 → 使用一組（作廢）→ ... → 剩餘 2 組時提醒 → 全部用完時強制重新產生
```

### 重新產生

- 使用者可在設定頁主動重新產生
- 重新產生時，舊的全部失效，產生新的 8 組
- 需確認密碼後才能操作

### 恢復碼剩餘不足時的處理

| 剩餘數量 | 行為 |
|----------|------|
| ≤ 2 | 登入後顯示警告提示，建議重新產生 |
| 0 | 登入後強制導向重新產生頁面 |

## 6. 資料表設計

### users 表新增欄位

```
users（既有表，新增以下欄位）
├── two_factor_secret          TEXT(nullable)       TOTP 密鑰（加密儲存）
├── two_factor_recovery_codes  TEXT(nullable)       恢復碼（加密 JSON 陣列）
└── two_factor_confirmed_at    TIMESTAMP(nullable)  2FA 設定確認時間
```

- `two_factor_secret`：以 `encrypt()` 加密儲存，解密後為 Base32 編碼的 TOTP 密鑰
- `two_factor_recovery_codes`：以 `encrypt()` 加密儲存，解密後為 JSON 陣列
- `two_factor_confirmed_at`：不為 null 表示使用者已完成 2FA 設定

### Migration 範例

```php
Schema::table('users', function (Blueprint $table) {
    $table->text('two_factor_secret')->nullable()->after('password');
    $table->text('two_factor_recovery_codes')->nullable()->after('two_factor_secret');
    $table->timestamp('two_factor_confirmed_at')->nullable()->after('two_factor_recovery_codes');
});
```

> 裝置相關資料表（`user_devices`）為獨立模組，詳見 [10009_UserDevice 裝置管理](10009_UserDevice裝置管理.md#6-資料表設計)。

## 7. 與現有機制的整合

### 2FA 的作用位置

2FA 僅作用於**互動式登入流程**（帳密驗證後、session 建立前），不影響四層安全架構的其他部分。

```
登入流程：
  帳密驗證 → [2FA 驗證] → Session 建立 → 進入受保護頁面
                ▲
                │
          僅在此處觸發
```

### 各認證方式的 2FA 適用性

| 認證方式 | 是否需要 2FA | 原因 |
|----------|-------------|------|
| **Session 認證（Web 登入）** | 是 | 互動式登入，2FA 在帳密驗證後觸發 |
| **Sanctum Bearer Token** | 間接 | 使用者在登入取得 token 時已通過 2FA，後續 API 請求不再觸發 |
| **Access Token** | 否 | Token 本身即已驗證憑證，用於 S2S/排程等非互動場景 |
| **Dev Impersonation** | 否 | 開發便利機制，非 production 環境，跳過 2FA |

### 與四層安全架構的關係

- 2FA 在**第三層身份層之前**，是登入流程的一部分，不是 middleware
- 四層安全架構（IP 限制 → X-API-KEY → 身份驗證 → 權限控制）不受 2FA 影響
- Access Token（`X-USER-ACCESS-TOKEN`）和 Dev Impersonation（`X-DEV-KEY`）繞過登入流程，因此不觸發 2FA
- 2FA 通過後取得的 Bearer Token 在整個 session/token 生命週期內有效，不需每次請求都驗 2FA

### 與 UserDevice 模組的關係

- **2FA 依賴 UserDevice**（必須） — 2FA 的所有觸發條件（新裝置、信任到期、閒置過久）皆需裝置記錄
- **UserDevice 不依賴 2FA**（獨立） — 裝置列表、遠端登出等功能可單獨運作
- 啟用 2FA 時，必須同時安裝 UserDevice 模組
- 移除 2FA 時，UserDevice 不受影響

- 四層安全架構完整說明：[10006_Portal 與 Api 安全機制](10006_Portal與Api安全機制.md)
- Sanctum 認證方式總覽：[10004_認證機制 Sanctum](10004_認證機制Sanctum.md)

## 8. 開發測試注意事項

### 快速關閉 2FA

```env
# .env
TWO_FACTOR_ENABLED=false
```

設為 `false` 即完全關閉 2FA 流程，所有登入直接帳密驗證，無需任何 2FA 操作。

### Dev Impersonation 跳過 2FA

Dev Impersonation（`X-DEV-KEY` + `X-DEV-USER-ID` / `X-DEV-USER-EMAIL`）直接透過 `CheckPortalAuthorization` 完成身份驗證，不經過登入流程，因此不觸發 2FA。開發/測試時可正常使用。

### Seeder 預設 2FA 資料

```php
// 測試用 Seeder，可預設 2FA secret 方便測試
User::factory()->create([
    'email' => 'admin@example.com',
    'two_factor_secret' => encrypt('JBSWY3DPEHPK3PXP'), // 測試用固定 secret
    'two_factor_recovery_codes' => encrypt(json_encode([
        'aaaaa-bbbbb',
        'ccccc-ddddd',
        'eeeee-fffff',
        'ggggg-hhhhh',
        'iiiii-jjjjj',
        'kkkkk-lllll',
        'mmmmm-nnnnn',
        'ooooo-ppppp',
    ])),
    'two_factor_confirmed_at' => now(),
]);
```

> **注意**：測試用的 TOTP secret `JBSWY3DPEHPK3PXP` 可在 Authenticator App 手動輸入，產生的驗證碼可用於測試 2FA 流程。

### 測試要點

| 測試項目 | 驗證內容 |
|----------|---------|
| 開關關閉 | `TWO_FACTOR_ENABLED=false` 時登入不觸發 2FA |
| 首次設定 | 開關開啟、未設定 2FA → 強制導向設定頁 |
| TOTP 驗證 | 正確碼通過、錯誤碼拒絕 |
| 恢復碼 | 使用後作廢、剩餘不足提醒 |
| 可信任裝置 | 信任期間免 2FA、到期後重新觸發 |
| 閒置裝置 | 裝置長期未使用後重新觸發 2FA |
| 密碼變更 | 清除所有信任裝置 |
| 新裝置 | 首次登入的裝置必須通過 2FA |
| Dev Impersonation | 不觸發 2FA |
| Access Token | 不觸發 2FA |

## 9. 未來擴充

### WebAuthn（指紋 / 硬體金鑰）

- 套件：`laragear/webauthn`
- 支援 FIDO2 標準，可使用指紋辨識器、YubiKey 等硬體金鑰
- 可作為 TOTP 的替代方案或並行方案
- 優先級：中（待使用者需求明確後再評估）

### SMS OTP

- 透過 SMS 發送一次性驗證碼
- 成本較高（需串接簡訊服務商）
- 安全性低於 TOTP（SIM swap 攻擊風險）
- 優先級：低，非優先項目

### 角色級 2FA 控管（進階需求）

- 管理員可對特定角色強制或豁免 2FA
- 例：`super_admin` 強制 2FA、`viewer` 豁免 2FA
- 需額外設定介面與邏輯
- 優先級：低，待系統規模擴大後再評估
