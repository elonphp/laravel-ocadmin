## 一、概述

### 1.1 技術選型

採用 [Spatie Laravel Permission](https://spatie.be/docs/laravel-permission) 作為權限管理基礎。

| 項目 | 說明 |
|------|------|
| 套件 | spatie/laravel-permission ^6.24 |
| Guard | 單一 Guard (`web`) |
| 角色命名 | Portal 前綴：`{portal}.{role_name}`（`super_admin`、`developer` 例外） |
| 權限命名 | 四段式：`{portal}.{module}.{resource}.{action}`，portal 段由 `config/portals.php` 的 `permission_prefix` 決定 |
| 萬用權限 | Spatie Wildcard Permission（`enable_wildcard_permission => true`） |
| 多語支援 | 角色與權限的 `display_name`、`note` 透過翻譯表實現 |
| 資料表前綴 | `acl_`（透過 `config/permission.php` 設定） |

### 1.2 權限類型

| 類型 | 說明 | 實現方式 |
|------|------|----------|
| **功能權限** | 能否執行某操作 | Spatie Permission |
| **資料權限** | 能看到哪些資料 | 組織層級 + Policy |

### 1.3 現有基礎設施

- [x] `spatie/laravel-permission` 套件安裝
- [x] `config/permission.php` 設定（`acl_` 前綴、wildcard 啟用）
- [x] ACL 資料庫遷移（含翻譯表）
- [x] 自訂 Model：`App\Models\Acl\Role`、`App\Models\Acl\Permission`（含多語翻譯）
- [x] `User` Model 已掛載 `HasRoles` trait
- [x] 角色 Seeder (`AclRoleSeeder`)
- [x] `developer` 角色：開發商最高權限，`Gate::before` 無條件放行
- [ ] 權限 Seeder
- [ ] 路由級權限檢查

## 二、角色設計

### 2.1 命名規範

**格式：`{portal}.{role_name}`**

```
hrm.hr_manager
 │      │
 │      └── 角色名稱（snake_case）
 └── Portal 前綴
```

例外：`super_admin`、`system`、`developer` 為全域角色，不帶 Portal 前綴。

> Portal 的定義與架構決策請見 [10001_Portal概述.md](10001_Portal概述.md)。

### 2.2 角色 ID 區段規劃

| ID 區段 | 用途 | 後台行為 |
|---------|------|---------|
| 1–20 | 系統保留角色 | 不可見、不可改（僅開發人員透過 DB/Seeder 調整） |
| 21–100 | 系統預設角色 | 可見、可指派給使用者、權限由 Gate::before 或 Seeder 控制 |
| 101+ | 使用者自建角色 | 完全可控（`auto_increment = 101`） |

判斷常數定義於 `Role` Model：`Role::SYSTEM_ROLES`（名稱清單，用於 `whereNotIn` 過濾）。

### 2.3 角色清單

#### 系統保留角色（id 1–20，後台不可見）

| id | name | 中文名稱 | 說明 |
|----|------|----------|------|
| 1 | `system` | 系統 | 系統操作標記角色（`users.username=service` 帳號賦予此角色） |
| 2 | `developer` | 開發者 | 開發商帳號，`Gate::before` 無條件放行 |
| 3–20 | — | — | 保留 |

#### 系統預設角色（id 21–100，後台可見）

| id | name | 中文名稱 | 說明 |
|----|------|----------|------|
| 21 | `super_admin` | 超級管理員 | 客戶方最高管理員，`Gate::before` 放行（`dev_only_permissions` 除外） |
| 22–100 | — | — | 保留供各專案 Seeder 定義 |

#### Portal 角色（id 101+，後台可自建）

各專案依需求定義，例如：

**HRM Portal（hrm.*）：**

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `hrm.hr_manager` | HR 主管 | 管理所有 HR 功能，含薪資、報表 |
| `hrm.hr_operator` | HR 管理員 | 日常 HR 操作：員工資料、出勤、請假 |
| `hrm.payroll_manager` | 薪資主管 | 薪資計算、薪資單管理 |
| `hrm.dept_manager` | 部門主管 | 審核下屬請假/出勤，檢視部門資料 |
| `hrm.employee` | 一般員工 | 查看個人資料、打卡、申請請假 |

**Ocadmin Portal（admin.*）：**

| 角色 name | 中文名稱 | 說明 |
|-----------|----------|------|
| `admin.store_manager` | 門市店長 | 擁有大部分 admin.* 權限 |
| `admin.operator` | 後台操作員 | 日常後台操作 |
| `admin.viewer` | 唯讀檢視者 | 僅擁有 `*.access` 權限 |

### 2.4 developer、system 與 super_admin 的分層

| | `system` | `developer` | `super_admin` |
|---|---|---|---|
| **ID 區段** | 1–20（系統保留） | 1–20（系統保留） | 21–100（系統預設） |
| **用途** | 系統操作標記 | 開發商帳號 | 客戶方最高管理員 |
| **Gate::before** | 無 | 無條件 `return true` | 放行所有權限（`dev_only_permissions` 除外） |
| **`can()` 快取** | — | 跳過（直接 `return true`） | 走角色組合快取 |
| **後台 UI 可見** | 不可見 | 不可見 | 可見 |
| **權限指派** | 無需權限 | 不需要（Gate 層放行） | 儲存時自動同步所有啟用權限 |
| **角色指派方式** | 僅 DB/Seeder | 僅 DB/Seeder | 後台可指派 |

**設計目的：** 將「開發商的全系統存取」與「客戶方的管理權限」分離。客戶的 `super_admin` 擁有大量權限但可控（例如不能看到開發者工具、Schema Parser 等），開發商的 `developer` 才是真正的最高權限。`system` 為系統操作標記角色，供系統自動執行的操作使用（如 `users.username=service` 帳號）。

### 2.4 跨角色指派

同一帳號可擁有多個角色：

```php
// HR 主管：管理功能 + 自己也是員工
$user->roles: ['hrm.hr_manager', 'hrm.employee']

// 部門主管：ESS + 部門管理
$user->roles: ['hrm.dept_manager', 'hrm.employee']

// 超級管理員：僅限 Ocadmin
$user->roles: ['super_admin']
```

## 三、權限設計

### 3.1 命名規範

**格式：`{portal}.{module}.{resource}.{action}`**

- **原則上 4 段**，使用點分隔，全小寫，snake_case
- resource 一律使用**單數**（如 `product`、`order`、`payment`，不用 `products`）
- module 多層用底線連接

| 段落 | 說明 | 範例 |
|------|------|------|
| `portal` | 系統入口（由 `config/portals.php` 的 `permission_prefix` 決定） | `hrm`, `admin` |
| `module` | 功能模組 | `ess`（自助）, `mss`（管理）, `team`（部門主管）, `catalog`, `order` |
| `resource` | 資源名稱 | `profile`, `employee`, `product`, `order`, `role` |
| `action` | 操作動作 | `access`, `modify`, `delete`, `approve`, `export` |

**非四段的例外情況：**

功能權限（CRUD 操作）固定 4 段，但用於**選單群組可見性控制**時，可使用較短的格式：

| 段數 | 格式 | 用途 | 範例 |
|------|------|------|------|
| 4 段 | `{portal}.{module}.{resource}.{action}` | 功能權限（原則） | `admin.master.supplier.access` |
| 2 段 | `{portal}.{module}` | 模組入口可見性 | `admin.master` |
| 3 段 | `{portal}.{module}.{resource}` | 資源入口可見性 | `admin.master.supplier` |

例如「主檔管理」選單群組設定 `permission: admin.master.access`，沒有此權限的角色整個主檔管理選單不可見，即使擁有其下子項的功能權限（見 [10011_選單機制.md](10011_選單機制.md)）。

### 3.2 萬用權限（Wildcard）

啟用 Spatie 原生 wildcard permission（`config/permission.php` 中 `enable_wildcard_permission => true`）。

使用 `*` 作為萬用字元：

```
# 單一資源的所有動作
hrm.mss.employee.*        → 符合 hrm.mss.employee.access、.modify、.delete、.export

# 整個模組的所有權限
hrm.mss.*.*               → 符合 hrm.mss.employee.*、hrm.mss.leave.*、hrm.mss.holiday.*

# 整個 Portal 的所有權限
hrm.*.*.*                  → 符合所有 hrm 權限

# 明確指定單一動作
hrm.ess.leave.modify       → 僅符合「請假申請」
```

> **注意**：不要建立 `*.all` 權限，使用 Spatie 原生 wildcard `*` 即可達成相同效果，無需額外的 Gate 邏輯。

### 3.3 命名範例

```

admin.catalog.product.modify        商品修改

admin.catalog.option.modify         選項修改
admin.catalog.category.access       分類檢視
admin.catalog.category.modify       分類修改

# Admin Portal — 訂單銷售
admin.sale.order.access             訂單檢視
admin.sale.order.modify             訂單修改
admin.sale.order.delete             訂單刪除
admin.sale.return.access            退貨檢視
admin.sale.return.modify            退貨處理
admin.sale.return.approve           退貨審核

# Admin Portal — 系統管理
admin.system_access.user.access     使用者檢視
admin.system_access.user.modify     使用者修改
admin.system_access.role.access     角色檢視
admin.system_access.role.modify     角色權限設定
admin.config.setting.access         系統設定檢視
admin.config.setting.modify         系統設定修改

# 欄位級權限（action 後加 :column）
admin.finance.payroll.access:salary  薪資金額
admin.finance.payroll.access:bonus   獎金金額
```

### 3.4 常用 Action

| Action | 說明 | 涵蓋範圍 |
|--------|------|----------|
| `menu` | 選單可見 | 純 UI 層，控制 sidebar 是否顯示該項目（見 [10011](10011_選單機制.md)） |
| `access` | 檢視 | 列表 + 單筆檢視（原 list + read），資料權限基礎 |
| `modify` | 修改 | 新增 + 修改（原 create + update） |
| `delete` | 刪除 | |
| `export` | 匯出 | |
| `approve` | 審核 | 適用於請假、出勤、退款等需審核流程 |

> **簡化原則**：能看就能列表（access），能改就能建（modify），刪除與審核獨立控管。

> **`menu` 與 `access` 拆開**：`menu` 只負責「選單是否顯示」，`access` 負責「能否進入該作業頁面 + 看資料」，兩者語意獨立、各自勾選。MenuComposer 用 `*.menu` 過濾，Controller `authorize()` 用 `*.access`。詳見 [10011_選單機制.md §1.1](10011_選單機制.md)。

> **`access` 是資料權限的基礎**：`modify`、`delete` 等權限建立在 `access` 之上——沒有 `access` 就進不了頁面、看不到資料，其他權限無從行使。因此指派權限時，給 `modify` 或 `delete` 必須同時給 `access`。

### 3.6 角色權限指派範例（Wildcard）

```php
// hrm.hr_manager — 使用萬用權限涵蓋所有動作
$hrManager->givePermissionTo([
    'hrm.mss.*.*',              // MSS 管理模組全部
    'hrm.team.*.*',             // 部門主管模組全部
    'hrm.ess.*.*',              // ESS 自助模組全部
]);

// hrm.hr_operator — 部分模組全開，部分限定
$hrOperator->givePermissionTo([
    'hrm.mss.employee.*',       // 員工管理全部動作
    'hrm.mss.department.access',
    'hrm.mss.designation.access',
    'hrm.mss.attendance.*',
    'hrm.mss.leave.*',
    'hrm.ess.*.*',              // ESS 自助全部
]);

// hrm.employee — 僅 ESS 個人權限
$employee->givePermissionTo([
    'hrm.ess.profile.access',
    'hrm.ess.profile.modify',
    'hrm.ess.attendance.modify',
    'hrm.ess.leave.access',
    'hrm.ess.leave.modify',
    'hrm.ess.payslip.access',
]);

// hrm.dept_manager — ESS + 部門管理
$deptManager->givePermissionTo([
    'hrm.team.employee.access',
    'hrm.team.attendance.access',
    'hrm.team.leave.approve',
]);
```

## 四、資料表結構

### 4.1 Spatie Permission 資料表

使用 `acl_` 前綴（透過 `config/permission.php` 設定）：

| 資料表 | 說明 |
|--------|------|
| `acl_permissions` | 權限 |
| `acl_roles` | 角色 |
| `acl_role_has_permissions` | 角色-權限關聯 |
| `acl_model_has_permissions` | 使用者-權限關聯（直接指派） |
| `acl_model_has_roles` | 使用者-角色關聯 |

### 4.2 翻譯表

| 資料表 | 說明 |
|--------|------|
| `acl_role_translations` | 角色多語翻譯（display_name, note） |
| `acl_permission_translations` | 權限多語翻譯（display_name, note） |

### 4.3 擴展欄位

`acl_permissions` 額外欄位：

| 欄位 | 類型 | 說明 |
|------|------|------|
| `is_active` | boolean, default true | 是否啟用 |

> 選單可見性透過獨立的 `*.menu` action 處理（見 [10011_選單機制.md §1.1](10011_選單機制.md)），不需在 `acl_permissions` 上加 `type` 欄位。權限分組顯示靠權限 name 的四段式結構解析（`{portal}.{module}` 前兩段）即可，不需 `parent_id` 樹狀結構。

`acl_roles` 額外欄位：

| 欄位 | 類型 | 說明 |
|------|------|------|
| `sort_order` | int, default 0 | 排序 |
| `is_active` | boolean, default true | 是否啟用 |

## 五、設定與安裝

### 5.1 config/permission.php（重點摘要）

```php
return [
    'models' => [
        'permission' => App\Models\Acl\Permission::class,  // 自訂 Model
        'role' => App\Models\Acl\Role::class,              // 自訂 Model
    ],

    'table_names' => [
        'roles' => 'acl_roles',
        'permissions' => 'acl_permissions',
        'model_has_permissions' => 'acl_model_has_permissions',
        'model_has_roles' => 'acl_model_has_roles',
        'role_has_permissions' => 'acl_role_has_permissions',
    ],

    'teams' => false,  // 不使用 teams 功能

    'enable_wildcard_permission' => true,  // 啟用萬用權限

    'cache' => [
        'expiration_time' => \DateInterval::createFromDateString('24 hours'),
        'key' => 'spatie.permission.cache',
        'store' => 'default',
    ],
];
```

### 5.2 config/portals.php

每個 Portal 定義 `role_prefix` 和 `permission_prefix`，分別用於角色與權限命名的第一段：

```php
return [
    'global'  => ['dir' => null],
    'ocadmin' => [
        'url_prefix'        => 'admin',
        'role_prefix'       => 'admin',       // 角色命名：admin.{role_name}
        'permission_prefix' => 'admin',       // 權限命名：admin.{module}.{resource}.{action}
        'dir'               => 'Ocadmin',
    ],
    'web'     => [
        'role_prefix'       => 'web',
        'permission_prefix' => 'web',
        'dir'               => 'Web',
    ],
];
```

- **`role_prefix`**：角色名稱的前綴段，如 `admin.order_operator`
- **`permission_prefix`**：權限名稱的前綴段，如 `admin.catalog.option.access`
- 兩者通常相同，但保持分開定義以便未來彈性調整

### 7.1 Policy

```php
// app/Policies/EmployeePolicy.php

class EmployeePolicy
{
    public function viewAny(User $user): bool
    {
        return $user->can('hrm.mss.employee.access')
            || $user->can('hrm.team.employee.access');
    }

    public function view(User $user, Employee $employee): bool
    {
        // HR 有權限可查看所有
        if ($user->can('hrm.mss.employee.access')) {
            return true;
        }

        // 查看自己
        if ($employee->user_id === $user->id) {
            return true;
        }

        // 部門主管查看下屬
        if ($user->can('hrm.team.employee.access')) {
            return $employee->department_id === $user->employee?->department_id;
        }

        return false;
    }
}
```

## 十、使用者權限快取

### 10.1 問題

Spatie 內建快取僅快取**全域權限定義表**（所有 permission 與 role 的對應關係），不快取個別使用者的權限。每個 request 仍需：

1. 載入使用者的 roles（1 次 DB query）
2. 載入使用者的 direct permissions（1 次 DB query）
3. In-memory 比對

系統規模擴大（使用者多、權限多）時，每個 request 固定 2 次 DB query 可優化。

### 10.2 方案：角色組合快取 + 全域版本號

**核心概念：** 將使用者的 role ids 排序後組成 cache key，快取該角色組合的所有權限名稱。相同角色組合的使用者共用同一份快取。

**Cache 結構：**

```
┌─────────────────────────┐     ┌──────────────────────────────────┐
│ key: role_perm_ver      │     │ key: role:1-3-11:v42             │
│ val: 42                 │     │ val: ['admin.catalog.product.ac… │
│ TTL: forever            │     │       'admin.master.supplier.ac… │
│                         │     │       ...]                       │
│ （全域版本號，所有角色   │     │ TTL: 7 days                      │
│   組合共用一個）         │     │                                  │
└─────────────────────────┘     │ （角色 1+3+11 的所有權限）        │
                                └──────────────────────────────────┘
```

### 10.3 查詢流程

```
使用者發 request
    │
    ├─ 1. 取 user 的 role_ids → [1, 3, 11]（從 DB 或 session）
    │
    ├─ 2. 取全域版本號 → Cache::get('role_perm_ver') → 42
    │
    ├─ 3. 組 cache key → "role:1-3-11:v42"
    │
    ├─ 4. Cache::get("role:1-3-11:v42")
    │     ├─ hit  → 拿到權限陣列（0 DB query）
    │     └─ miss → 查 DB 建立權限陣列，寫入 cache（TTL 7 天）
    │
    └─ 5. $user->can() 在 in-memory 比對
```

**每 request 成本：** 2 次 cache get（版本號 + 權限），cache hit 時 0 次 DB query。

### 10.4 快取失效：全域版本號

角色的權限被修改時，**只遞增版本號，不主動重建快取**：

```php
// 角色權限變更時（如 RoleController 儲存權限後）
Cache::increment('role_perm_ver');
```

- 版本號從 42 → 43
- 所有 request 組出的 key 自動從 `v42` 變 `v43`
- 全部 cache miss，各自在下一次 request 時從 DB 重建
- 舊版 `v42` 的 key 無人再讀，靠 TTL（7 天）自然回收

**不在變更當下重建的原因：** 管理員設定權限時可能連續修改多次，若每次都重建會造成不必要的 IO。Lazy rebuild（等 request 進來時才建）更合理。

### 10.5 實作要點

**User Model 覆寫 `can()`：**

```php
// app/Models/User.php

public function can($ability, $arguments = [])
{
    // Policy 檢查（有 arguments 時走原生 Gate 邏輯）
    if (!empty($arguments)) {
        return parent::can($ability, $arguments);
    }

    // developer 跳過快取（Gate::before 也會放行，這裡提前 return 省掉 cache lookup）
    if ($this->hasRole('developer')) {
        return true;
    }

    $permissions = $this->getCachedPermissions();

    return in_array($ability, $permissions);
}

protected function getCachedPermissions(): array
{
    $roleIds = $this->roles->pluck('id')->sort()->implode('-');
    $ver = Cache::get('role_perm_ver', 1);
    $key = "role:{$roleIds}:v{$ver}";

    return Cache::remember($key, now()->addDays(7), function () {
        return $this->getPermissionsViaRoles()
            ->pluck('name')
            ->toArray();
    });
}
```

**角色權限變更時遞增版本號：**

```php
// 角色權限 sync 後
$role->syncPermissions($permissions);
Cache::increment('role_perm_ver');
```

**版本號初始化與 Gate::before（AppServiceProvider boot）：**

```php
// Cache::add 僅在 key 不存在時寫入，每次 boot 呼叫無副作用
Cache::add('role_perm_ver', 1);

// developer 角色無條件放行（真正的最高權限）
// super_admin 不在此處理，走正常權限流程
Gate::before(fn ($user, $ability) => $user->hasRole('developer') ? true : null);
```

### 10.6 TTL 設計

| Cache entry | TTL | 原因 |
|-------------|-----|------|
| `role_perm_ver`（版本號） | forever | 關鍵資料，不可意外過期 |
| `role:{ids}:v{ver}`（權限集合） | 7 天 | 當前版本實質 forever（持續命中）；舊版靠 TTL 自動回收 |

### 10.7 注意事項

- **直接指派權限（`model_has_permissions`）：** 此快取僅涵蓋透過角色取得的權限。若有使用直接指派，需在 cache key 中加入 user id 或另行處理
- **角色變更 vs 權限變更：** 使用者的角色被更換時，其 role_ids 組合改變，自然組出不同 cache key，不需要額外處理版本號
- **Cache driver：** 此方案不依賴 Cache Tags，任何 driver（file、database、Redis、Memcached）皆可使用
- **停用帳號：** 緊急封鎖惡意使用者應透過停用帳號（`is_active = false`，auth middleware 攔截），不應依賴權限快取失效

### 10.8 與 Spatie 內建快取的比較

| | Spatie 內建 | 角色組合快取 |
|---|---|---|
| **快取內容** | 全域權限定義表 | 角色組合 → 權限名稱集合 |
| **每 request DB query** | 2 次（user→roles, user→permissions） | 0 次（cache hit 時） |
| **快取命中率** | 100%（全域一份） | 高（同角色組合共用，實務上組合有限） |
| **失效機制** | 權限/角色定義異動時自動清 | 全域版本號遞增 |
| **Cache driver 限制** | 無 | 無 |

## 十二、單控制器多選單入口

### 12.1 情境

同一筆資料（如供應商）需要從不同選單入口操作：

- **主檔管理 > 供應商**：`/admin/master/suppliers`
- **庫存管理 > 供應商**：`/admin/inventory/suppliers`

兩個入口使用**同一個 Controller**（`SupplierController`），但使用者從哪個入口進來，返回按鈕、列表連結、分頁連結就要指回對應的路徑。

### 12.2 路由：相同 Controller 掛在不同前綴

```php
// 主檔管理 > 供應商
Route::prefix('master')->name('master.')->group(function () {
    Route::prefix('suppliers')->name('supplier.')->group(function () {
        Route::get('/', [SupplierController::class, 'index'])->name('index');
        Route::get('/list', [SupplierController::class, 'list'])->name('list');
        Route::get('/create', [SupplierController::class, 'create'])->name('create');
        // ...
    });
});

// 庫存管理 > 供應商（共用同一個 Controller）
Route::prefix('inventory')->name('inventory.')->group(function () {
    Route::prefix('suppliers')->name('supplier.')->group(function () {
        Route::get('/', [SupplierController::class, 'index'])->name('index');
        Route::get('/list', [SupplierController::class, 'list'])->name('list');
        Route::get('/create', [SupplierController::class, 'create'])->name('create');
        // ...
    });
});
```

產生兩組 route name：

| 入口 | route name 前綴 |
|------|----------------|
| 主檔管理 | `lang.ocadmin.master.supplier` |
| 庫存管理 | `lang.ocadmin.inventory.supplier` |

### 12.3 Controller：用 `routePrefix()` 動態推算

Controller 不寫死路由名稱，而是從當前 request 的 route name 反推前綴：

```php
/**
 * 從當前 route name 反推前綴
 * e.g. 'lang.ocadmin.master.supplier.index' → 'lang.ocadmin.master.supplier'
 *      'lang.ocadmin.inventory.supplier.edit' → 'lang.ocadmin.inventory.supplier'
 */
protected function routePrefix(): string
{
    return preg_replace('/\.[^.]+$/', '', request()->route()->getName());
}
```

所有 URL 都透過前綴動態產生：

```php
public function index(Request $request): View
{
    $prefix = $this->routePrefix();

    $data['list_url']  = route("{$prefix}.list");
    $data['index_url'] = route("{$prefix}.index");
    $data['add_url']   = route("{$prefix}.create");
    // ...
}

public function create(): View
{
    $prefix = $this->routePrefix();

    $data['save_url'] = route("{$prefix}.store");
    $data['back_url'] = route("{$prefix}.index");  // 返回按鈕回到來源列表
    // ...
}
```

### 12.4 View：不需任何修改

View 只使用 Controller 傳入的 `$back_url`、`$list_url` 等變數，不關心具體路徑。

### 12.5 效果

| 使用者操作 | 產生的 URL |
|-----------|-----------|
| 從主檔管理進入，點新增 | `/admin/master/suppliers/create` |
| 按返回 | `/admin/master/suppliers` |
| 從庫存管理進入，點新增 | `/admin/inventory/suppliers/create` |
| 按返回 | `/admin/inventory/suppliers` |

### 12.6 注意事項

- 兩組路由的 action name（`index`、`create`、`store`、`edit`、`update`、`destroy`）必須一致
- Route Model Binding 的參數名也要一致（如 `{organization}`）
- 權限控制在 Controller constructor 中處理，兩個入口可共用或分別設定權限

## 十三、設計決策說明（為何選單數）

§3.1 規定 resource 段一律用單數（`product`、`order`、`user`，不用複數）。
這個決策有時會被質疑——畢竟 RESTful 路由用複數（`/products`、`/users`），
業界 IAM 命名也常見複數（GCP IAM `compute.instances.get`、Spatie 文件範例
`edit articles`）。本節說明為何本規範刻意選**單數**：

### 13.1 跟 Eloquent Model 命名對齊

Laravel Model 用單數（`User`、`Product`、`Order`），permission 跟著單數，
跨 layer 一致：

```
Model:      App\Models\Product
Policy:     App\Policies\ProductPolicy
Permission: admin.catalog.product.access     ← 看到 permission name 立刻知道對應 Model
Route URL:  /admin/catalog/products          ← URL 走 RESTful 複數慣例（不衝突）
```

URL 跟 permission 各走各的慣例，不互相牽動。維護者習慣這個分工後，看 permission
就知道對應哪個 class。

### 13.2 英文複數變化複雜，單數最簡單

部分名詞複數變化規則特殊：

| 單數 | 複數 |
|---|---|
| `category` | `categories` |
| `taxonomy` | `taxonomies` |
| `status` | `statuses` |
| `child` | `children` |
| `phenomenon` | `phenomena` |
| `analysis` | `analyses` |

選單數 = 永遠不用想複數規則，跨整個系統 token 一致。

### 13.3 Wildcard 語意更直覺

```
admin.catalog.product.*    讀作「對 product 資源的所有動作」     ← 直覺
admin.catalog.products.*   讀作「對 products 集合的所有動作」？  ← 含混：是集合操作還是逐筆？
```

單數版用 wildcard 時，語意自然指向「對該資源類型的所有動作」，不會有「集合 vs 個體」
的歧義。

### 13.4 避免 list/show 的二元區分

複數版命名容易引導出「集合操作 vs 個體操作」二元化：

```
products.list   看清單
products.show   看單筆
```

但 spatie / Laravel ACL 沒這層區分，看清單跟看單筆通常用同一個 `access` 權限。
單數版自然不會落入這個陷阱：

```
product.access   一個權限管「能否進該資源的存取頁面」（含清單與單筆）
```

### 13.5 Action 段（第 4 段）天生單數

`access`、`modify`、`delete`、`approve`、`export`、`pay` 都是動詞，動詞沒複數
變化問題。第 4 段一律單數沒爭議；既然第 4 段單數，第 3 段也跟著單數，整個 token
讀起來最一致。

### 13.6 業界並非壓倒性偏向某一方

實際上業界混用，並無公認標準：

| 系統 | 慣例 |
|---|---|
| AWS IAM | 混用（`s3:GetObject` 單數、`ec2:DescribeInstances` 複數） |
| GCP IAM | 偏複數（`compute.instances.get`） |
| Spatie 官方文件 | 偏複數（範例 `edit articles`） |
| Laravel Gate / Policy | 無強制慣例 |

本規範選擇單數是**內部一致性決定**（與 Model / Policy 對齊、與英文不規則複數
脫鉤），不是因為「業界都這麼做」。
