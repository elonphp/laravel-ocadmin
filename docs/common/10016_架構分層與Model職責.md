## 設計原則

1. **簡單優先，按需抽層**：預設只有 Controller + Model 兩層，其他層要有明確觸發條件才開。
2. **YAGNI**：不為了「未來可能需要」而提前抽象。事實證明重構成本比過度設計成本低得多。
3. **抽層的唯一觸發條件是「共用需求」**：同一段邏輯被兩個以上地方使用，且無法靠 Eloquent/Scope 自然共用，才抽。
4. **不採用 Repository**：Laravel 的 Eloquent 本身已是 Active Record + Query Builder，Repository 在這個框架下幾乎永遠是多餘的一層。細節見〈[為什麼不採用 Repository](#為什麼不採用-repository)〉。

## 分層總覽

| 層 | 位置 | 用途 | 預設是否開啟 |
|----|------|------|--------------|
| Controller | `app\Portals\{Portal}\Controllers\` | HTTP 入口、Request/Response、Portal 邊界 | 永遠 |
| Model | `app\Models\` | Active Record、關聯、Scope、defaults | 永遠 |
| Service | `app\Services\` | 跨 Portal 共用的業務動作（create/update/交易流程） | 按需 |
| Integrations | `app\Integrations\` | 外部服務（SMS、Email、金流、物流 API） | 按需 |
| Portal Service | `app\Portals\{Portal}\Services\` | Portal 專屬的複雜編排邏輯 | 最後才開 |

**不存在的層**：`app\Repositories\`。舊專案裡負責 `save()`、`setDefaultData()` 的 Repository，新專案一律改寫為 `app\Services\{Entity}Service`。

## 預設流程：Controller 直接操作 Model

大多數 CRUD 不需要 Service，Controller 直接使用 Eloquent：

```php
// 列表
$orders = Order::query()
    ->filterByPhone($filter_data['filter_phone'] ?? null)
    ->paginate();

// 建立
$order = new Order();
$order->fill($request->validated());
$order->save();

// 更新
$order->fill($request->validated());
$order->save();
```

當發生以下情況之一，再往上抽層：

- 同一業務動作在兩個以上 Portal 出現 → 抽 `app\Services\{Entity}Service`
- 需要呼叫外部 API（簡訊、金流、Email）→ 抽 `app\Integrations\{Vendor}\{Vendor}Client`
- 單一 Controller 方法超過 ~150 行編排邏輯 → 抽 Service 或 Portal Service

## Model 職責

### Model 應該做的事

| 職責 | 做法 |
|------|------|
| 欄位定義與型別 | `$attributes`（預設值）、`$casts`、`$fillable` / `$guarded` |
| 關聯 | `hasMany()`、`belongsTo()`、`belongsToMany()` 等 |
| 查詢 Scope | `scopeFilterByXxx()`，處理複雜/跨欄位/跨關聯查詢 |
| 表單預設值 | `defaults()` 靜態方法，供 Controller `create()` 使用 |
| Accessor / Mutator | 欄位顯示/儲存轉換（用 `Attribute::make()`） |

### Model 不應該做的事

| 不該做 | 該放哪裡 |
|--------|----------|
| 跨 Model 的業務流程（建單同時扣庫存、寫 log） | `app\Services\` |
| HTTP / Request 處理 | Controller |
| 呼叫外部服務（送簡訊、刷卡） | `app\Integrations\` |
| Portal 專屬條件（如 `brand_id` 隔離） | Controller 或 Middleware |

## 複雜查詢：Model Scope

`OrmHelper::prepare()` 搭配 `filterData()` 可處理**標準欄位查詢**（`filter_name`、`equal_status` 等）。當查詢參數**無法直接對應到單一資料表欄位**時，需要 Model Scope：

| 情境 | 說明 |
|------|------|
| 多欄位 OR 搜尋 | 輸入「電話」需同時搜尋 `mobile` 和 `telephone` |
| 跨關聯查詢 | 依訂單商品名稱搜尋訂單 |
| 特殊值轉換 | `withoutV` 要轉成 `status_code <> 'void'` |
| 彈性格式解析 | 日期輸入支援多種格式（`20260301`、`2026-03-01`、`20260301-20260331`） |

這些邏輯屬於**領域知識**，不應分散在各 Portal Controller，應集中於 Model Scope。

### 什麼時候需要 Scope

```
filter_phone → mobile LIKE '%...%' OR telephone LIKE '%...%'   ← 一對多欄位，需 Scope
filter_name  → name LIKE '%...%'                               ← 標準，OrmHelper 直接處理
equal_status → status = '...'                                  ← 標準，OrmHelper 直接處理
filter_product_name → whereHas('orderProducts', ...)           ← 跨關聯，需 Scope
filter_delivery_date → BETWEEN / >= / <=（彈性格式）           ← 格式解析，需 Scope
```

判斷標準：**參數名稱無法直接對應到單一資料表欄位** → 使用 Model Scope。

### 命名規則

| 規則 | 範例 |
|------|------|
| 前綴一律用 `scope` | `scopeFilterByPhone` |
| 語意對應使用者輸入的概念 | `filterByPhone`（而非 `filterByMobileOrTelephone`） |
| 空值時直接 return | `if (empty($value)) return;` |

### Scope 範例

**多欄位 OR 搜尋**（改用 `OrmHelper::filterOrEqualColumn()` 支援萬用字元 `*0912`、`=0912345678`）：

多欄位 OR 的模式：第一個欄位直接呼叫，後續欄位各自包一層 `orWhere(function)` 再呼叫（因為 `filterOrEqualColumn` 內部使用 `->where()`，不能直接串 OR）：

```php
public function scopeFilterByPhone(Builder $query, ?string $phone): void
{
    if (empty($phone)) return;
    $phone = preg_replace('/[\s\-]/', '', $phone);
    $query->where(function ($q) use ($phone) {
        OrmHelper::filterOrEqualColumn($q, 'filter_mobile', $phone);
        $q->orWhere(function ($sub) use ($phone) {
            OrmHelper::filterOrEqualColumn($sub, 'filter_telephone', $phone);
        });
    });
}

public function scopeFilterByKeyname(Builder $query, ?string $keyname): void
{
    if (empty($keyname)) return;
    $query->where(function ($q) use ($keyname) {
        OrmHelper::filterOrEqualColumn($q, 'filter_personal_name', $keyname);
        $q->orWhere(function ($sub) use ($keyname) {
            OrmHelper::filterOrEqualColumn($sub, 'filter_shipping_personal_name', $keyname);
        });
        $q->orWhere(function ($sub) use ($keyname) {
            OrmHelper::filterOrEqualColumn($sub, 'filter_shipping_company', $keyname);
        });
        $q->orWhere(function ($sub) use ($keyname) {
            OrmHelper::filterOrEqualColumn($sub, 'filter_payment_company', $keyname);
        });
    });
}
```

**跨關聯查詢**：

```php
public function scopeFilterByProductName(Builder $query, ?string $name): void
{
    if (empty($name)) return;
    $query->whereHas('orderProducts', function ($q) use ($name) {
        OrmHelper::filterOrEqualColumn($q, 'filter_name', $name);
    });
}
```

**彈性日期格式**：

```php
// 支援多種輸入格式：20260301、2026-03-01、20260301-20260331、20260301-
public function scopeFilterByDeliveryDate(Builder $query, ?string $dateStr): void
{
    if (empty($dateStr)) return;
    $sql = DateHelper::parseDateToSqlWhere('delivery_date', $dateStr);
    if ($sql) $query->whereRaw("({$sql})");
}
```

**特殊值轉換**：

```php
public function scopeFilterByStatusCode(Builder $query, ?string $statusCode): void
{
    if (empty($statusCode)) return;
    if ($statusCode === 'withoutV') {
        $query->where('status_code', '<>', 'void');
    } else {
        OrmHelper::filterOrEqualColumn($query, 'filter_status_code', $statusCode);
    }
}
```

### 安全性：白名單的必要性

`OrmHelper::applyFilters()` 的實作是「欄位存在於資料表就套用」，若不加白名單，使用者可自行在 URL 加入 `?equal_customer_id=5&filter_is_closed=0`，這在**前台對外 Portal 是明確的安全漏洞**。

解決方式為 `filterData()` 白名單（完整說明見 [00003_Ocadmin程式規範.md](00003_Ocadmin程式規範.md#filterdata-白名單機制)）：

```php
$filter_data = $this->filterData($request, [
    'filter_code',
    'filter_phone',          // 進 Scope，不進 OrmHelper
    'filter_delivery_date',  // 進 Scope，不進 OrmHelper
    'equal_status_code',
]);
```

**注意：傳空陣列 `[]` 不等於全開放**，反而是最嚴格——只剩共用參數（`sort, order, page` 等），所有 `filter_*` / `equal_*` 都不會進來。後台內部頁面如需全開放，請用 `$request->query()` 並加注解說明原因。**前台對外 Portal 一律必須使用白名單。**

**Scope 處理的參數，在 `OrmHelper::prepare()` 之前必須從 `$filter_data` 移除**，避免 OrmHelper 重複套用：

```php
$query->filterByPhone($filter_data['filter_phone'] ?? null);
unset($filter_data['filter_phone']);
```

### Controller 整合方式

```php
protected function getList(Request $request): string
{
    // 1. 白名單（含 Scope 用到的參數）
    $filter_data = $this->filterData($request, [
        'filter_phone',
        'filter_keyname',
        'filter_delivery_date',
        'filter_product_name',
        'filter_status_code',
        'filter_code',
    ]);

    $filter_data['sort']  = $request->query('sort', 'id');
    $filter_data['order'] = $request->query('order', 'desc');

    // 2. Scope 處理複雜查詢
    $query = Order::query()
        ->filterByPhone($filter_data['filter_phone'] ?? null)
        ->filterByKeyname($filter_data['filter_keyname'] ?? null)
        ->filterByDeliveryDate($filter_data['filter_delivery_date'] ?? null)
        ->filterByProductName($filter_data['filter_product_name'] ?? null)
        ->filterByStatusCode($filter_data['filter_status_code'] ?? null);

    unset(
        $filter_data['filter_phone'],
        $filter_data['filter_keyname'],
        $filter_data['filter_delivery_date'],
        $filter_data['filter_product_name'],
        $filter_data['filter_status_code'],
    );

    // 3. OrmHelper 處理剩餘標準欄位
    OrmHelper::prepare($query, $filter_data);

    // 4. 分頁
    $orders = OrmHelper::getResult($query, $filter_data);
    $orders->withPath(route('lang.ocadmin.sale.orders.list'));
}
```

Scope 用到的參數**仍需列入白名單**，否則該參數在進入 Controller 前就已被過濾掉：

```php
// ❌ 錯誤
$filter_data = $this->filterData($request, ['filter_code']);
$query->filterByPhone($filter_data['filter_phone'] ?? null); // 永遠是 null

// ✓ 正確
$filter_data = $this->filterData($request, ['filter_code', 'filter_phone']);
```

### 多 Portal、多品牌環境

Scope 定義在 Model，是整個系統共用的領域知識。**各 Portal 自行組合需要的 Scope**，不需要 Repository 中介：

```php
// Ocadmin Portal（後台）：完整搜尋欄位
$query = Order::query()
    ->filterByPhone($filter_data['filter_phone'] ?? null)
    ->filterByKeyname($filter_data['filter_keyname'] ?? null)
    ->filterByDeliveryDate($filter_data['filter_delivery_date'] ?? null)
    ->filterByProductName($filter_data['filter_product_name'] ?? null);

// PosCateringV3 Portal（POS）：只需少數查詢條件
$query = Order::query()
    ->filterByDeliveryDate($filter_data['filter_delivery_date'] ?? null);
```

**品牌隔離不放進 Scope**，因為它是 Portal 的隱性條件，而非使用者輸入的搜尋條件：

```php
// Portal Controller 自行加
$query = Order::query()
    ->where('brand_id', $this->currentBrand()->id)
    ->filterByDeliveryDate($filter_data['filter_delivery_date'] ?? null);
```

## Model 預設值（defaults）

預設值依複雜度分層放置，**不集中在單一 `prepareData()` 方法**（`prepareData` 與 `OrmHelper::prepare()` 撞詞，語意不清）：

| 預設值類型 | 放置位置 | 說明 |
|-----------|---------|------|
| 簡單欄位預設（空字串、0、enum 初始值） | `Model::$attributes` | Laravel 原生，自動套用於新實例 |
| 表單初始值（含關聯選項、顯示預設） | `Model::defaults()` 靜態方法 | 命名語意明確，與查詢邏輯無關 |
| 需要 DB 查詢的預設（如預設門市） | `app\Services\{Entity}Service::newModel()` | 需 DB 依賴時往 Service 放，不污染 Model |

```php
class Order extends Model
{
    // 簡單欄位預設（Laravel 原生）
    protected $attributes = [
        'status_code' => 'pending',
        'quantity'    => 1,
        'is_visible'  => true,
    ];

    // 表單初始值（靜態方法，供 Controller create() 使用）
    public static function defaults(): array
    {
        return [
            'status_code'    => 'pending',
            'delivery_type'  => 'pickup',
            'payment_method' => 'cash',
        ];
    }
}
```

```php
// Controller create()
public function create(): View
{
    $data['order'] = Order::defaults();
    return view('ocadmin::sale.order.form', $data);
}

// Controller edit()（從 DB 取得，不需 defaults）
public function edit(Order $order): View
{
    $data['order'] = $order;
    return view('ocadmin::sale.order.form', $data);
}
```

## Service 層：何時抽、怎麼寫

### 抽 Service 的觸發條件

滿足以下**任一**條件才抽 Service，否則 Controller 直接操作 Model：

| 觸發條件 | 範例 |
|---------|------|
| A. 兩個以上 Portal 共用同一業務動作 | Ocadmin 與 POS 都能建單 → `OrderService::create()` |
| B. 單一動作涉及多 Model 且有交易/狀態機 | 建單同時寫 `orders`、`order_products`、扣 `product_stocks` |
| C. 業務邏輯需要脫離 HTTP context 測試 | 排程任務、Queue Job 呼叫同一業務動作 |

**不要為了「分層乾淨」而抽**——Service 如果只是把參數轉交 Eloquent，就是多餘的一層。

### 位置與命名

- 位置：`app\Services\`（flat 結構，不分 Portal）
- 命名：按**業務實體**命名，不按 Controller 命名
  - ✓ `OrderService`、`ProductService`、`InventoryService`
  - ✗ `OrderController 的 Service` 不叫 `OcadminOrderService`

```
app\Services\
    OrderService.php
    ProductService.php
    InventoryService.php
```

### create / update / save 共用模式

多數情況下新增/編輯只差 id、邏輯大量重複，抽一個共用 `save()` 處理共用欄位與關聯：

```php
namespace App\Services;

use App\Models\Order;
use App\Models\User;
use Illuminate\Support\Facades\DB;

class OrderService
{
    public function create(array $data, User $user): Order
    {
        return DB::transaction(fn() => $this->save(new Order(), $data, $user));
    }

    public function update(Order $order, array $data, User $user): Order
    {
        return DB::transaction(fn() => $this->save($order, $data, $user));
    }

    protected function save(Order $order, array $data, User $user): Order
    {
        // 共用欄位填值
        $order->fill($data);

        // 新增/編輯不對稱的部分，用 $order->exists 判斷
        if ($order->exists) {
            $order->updated_by = $user->id;
        } else {
            $order->code       = $this->generateCode();
            $order->created_by = $user->id;
        }

        $order->save();

        // 共用關聯 sync
        $order->products()->sync($data['products'] ?? []);

        return $order;
    }
}
```

Controller 使用：

```php
public function store(StoreOrderRequest $request, OrderService $orderService)
{
    $order = $orderService->create($request->validated(), $request->user());
    return redirect()->route('orders.edit', $order);
}
```

**原則**：`save()` 只處理「新增與編輯都要做」的事；新增/編輯各自專屬的邏輯（產 code、audit log、寄通知）留在 `create()`/`update()` 外層。不要為了 DRY 把所有東西塞進 `save()`，一旦出現 `if ($order->exists)` 超過 2 處就該把該邏輯拉回外層。

### Service 不該做的事

- **不要包 Eloquent**：不要寫 `findById($id)`、`findAll()` 這種 pass-through 方法，直接用 `Order::find($id)`。
- **不要做 HTTP**：`$request`、`redirect()`、`response()` 留在 Controller。
- **不要呼叫外部 API**：簡訊、金流、Email 屬於 Integrations。Service 若需要通知，應該注入 Integration Client。

## Integrations 層：外部服務

### 什麼東西放這裡

- 簡訊（Every8d、Mitake）
- Email（Mailgun、SendGrid）
- 金流（ECPay、NewebPay）
- 物流 API（黑貓、宅配通）
- 其他第三方 API

### 位置與命名

```
app\Integrations\
    Sms\
        SmsClient.php       // 介面或抽象類
        Every8dClient.php
        MitakeClient.php
    Ecpay\
        EcpayClient.php
    Mail\
        MailClient.php
```

使用 `Client` 或 `Gateway` 後綴，避免與 `Service` 混淆。

### 為什麼跟 Service 分開

- **Integrations 是「對外溝通」**，Service 是「內部業務」。
- 兩者壽命不同：外部 API 簽約/改版週期長，業務邏輯日日改。放一起會互相污染。
- Service 需要送簡訊時，注入 `SmsClient`，Service 本身不知道用哪家廠商。

```php
class OrderService
{
    public function __construct(private SmsClient $sms) {}

    public function confirm(Order $order): void
    {
        $order->status_code = 'confirmed';
        $order->save();
        $this->sms->send($order->customer->mobile, "訂單 {$order->code} 已確認");
    }
}
```

## Portal Service：預設不開

### 為什麼預設不開

Portal Service（`app\Portals\{Portal}\Services\`）在舊設計中是「1:1 對應 Controller」。**99% 會變成 pass-through**：Controller 呼叫 Service，Service 只是把參數轉交 Model 或 Domain Service，沒有實質價值，還讓新人多讀一層才能理解流程。

**Controller 本身就是 Portal 邊界**，已經承擔了「這個 Portal 的入口」這個職責。不需要再包一層。

### 什麼時候才開

**同時**滿足以下條件才考慮：

1. Controller 單一方法超過 ~150 行，且這段邏輯無法抽成跨 Portal 共用的 `app\Services\`。
2. 該 Portal 有獨特的編排流程（例如 POS 結帳有離線快取、Ocadmin 沒有）。
3. 需要在 HTTP context 外重用（例如 POS 的結帳流程也要被離線同步 Job 呼叫）。

只滿足 1 時，先抽成 Controller 內的 private method；只滿足 2 時，先看能否透過策略模式放進 `app\Services\`；三者都滿足，才開 Portal Service。

## 為什麼不採用 Repository

### Eloquent 已涵蓋 Repository 的職責

Repository Pattern 來自 Java / .NET 生態，目的是在 ORM 外再包一層，讓業務層不依賴 ORM。但 **Laravel Eloquent 本身就同時是 Active Record + Query Builder**：

- 寫入：`$order->save()`、`$order->delete()`
- 查詢：`Order::where(...)->get()`
- 領域查詢邏輯：Model Scope（`Order::filterByPhone(...)`）
- 交易：`DB::transaction(fn() => ...)`

Repository 在 Laravel 裡通常只是「把 `Order::find($id)` 包裝成 `$repo->find($id)`」，除了增加一層間接性，沒有帶來任何能力。

### 常見的三層誤區

```
❌ Controller → Service → Repository → Eloquent
```

這個疊法會出現：
- Service 方法多半只是轉發給 Repository
- Repository 方法多半只是轉發給 Eloquent
- 新增一個查詢要改三處
- 同一個 Controller 有時走 Service、有時直接走 Repository，沒有一致規則

### 取代方案

```
✓ Controller → Model（簡單 CRUD，預設）
✓ Controller → Service → Model（跨 Portal 共用業務動作）
✓ Service → IntegrationClient（呼叫外部 API）
```

**寫入/業務 → Service**，**查詢 → Model Scope**。兩者都建立在 Eloquent 上，不需要 Repository 中介。

### 舊系統 `OrderRepository::resetQueryData()` 的問題

舊系統用 Repository 做複雜查詢前處理：

```php
public function resetQueryData($data)
{
    if (!empty($data['filter_phone'])) {
        $data['andOrWhere'][] = [
            'filter_mobile' => $data['filter_phone'],
            'filter_telephone' => $data['filter_phone'],
        ];
        unset($data['filter_phone']);
    }
    return $data;
}
```

| 缺點 | 說明 |
|------|------|
| 型別不安全 | `$data` 是無型別的可變 array，輸入輸出契約不明確 |
| 隱性結構 | `andOrWhere`、`whereRawSqls` 等內部結構對呼叫端不透明 |
| 難以測試 | 需要帶入整個 `$data` array 才能測試單一規則 |
| 多 Portal 不友善 | 邏輯藏在 Repository，各 Portal 需繞過或重寫 |

新系統用 Model Scope 取代：

| 優點 | 說明 |
|------|------|
| 方法簽名明確 | `filterByPhone(?string $phone)` 一眼知道輸入型別 |
| 可獨立測試 | 每個 Scope 可單獨測試 |
| 多 Portal 共用 | Scope 在 Model，任何 Portal 直接呼叫 |
| 可組合 | 各 Portal 自由選擇需要哪些 Scope |

## 分層決策速查表

| 情境 | 做法 |
|------|------|
| 單 Portal 列表查詢 | Controller + Model Scope |
| 多 Portal 共用列表查詢 | Controller + Model Scope（各 Portal 自己組合需要的 Scope） |
| 單 Portal 簡單建單/編輯 | Controller + Eloquent（`$order->fill()->save()`） |
| 多 Portal 共用建單/編輯 | Controller + `app\Services\OrderService` |
| 跨 Model 的交易流程（建單扣庫存） | `app\Services\` |
| 送簡訊 / Email / 刷卡 | `app\Integrations\{Vendor}\{Vendor}Client` |
| Service 需要通知使用者 | Service 注入 Integration Client |
| 表單預設值（純欄位） | `Model::$attributes` 或 `Model::defaults()` |
| 表單預設值（需 DB 查詢） | `app\Services\{Entity}Service::newModel()` |
| Portal 的 Controller 超肥且邏輯獨特 | 先抽 private method；仍不夠才開 `app\Portals\{Portal}\Services\` |
| 品牌隔離、Portal 權限條件 | Controller 或 Middleware 加 where，不進 Scope |
